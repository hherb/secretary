## (3) Open decisions and risks

- **Scope was deliberately tight.** Trash view + Settings are out of D.5.3 by the user's choice — do not "finish" them into this PR. They are D.5.4.
- **The manual GUI smoke is the one outstanding pre-merge gate** (see §1). Everything else (host suites + app compile) is green and automated. This mirrors the D.1.14/D.1.15 posture where the interactive GUI smoke is manual-only ([#161](https://github.com/hherb/secretary/issues/161)-style).
- **macOS interaction diverges from iOS by design.** iOS uses swipe actions; macOS uses context menus + toolbar (idiom A, user-approved). If a future reviewer expects swipe parity, that's intentional — swipe-to-delete is undiscoverable with a mouse.
- **Tag deletion is clear-to-empty**, not a per-row delete button (tags are plain `[String]` with no identity — a `ForEach(indices)` delete-by-index is crash-prone). Emptied tags are dropped on save (the VM already filters them). Fields *do* have a per-row delete (identity-safe, keyed on `EditableField.id`). If you add per-row tag delete later, give tags a stable identity first.
- **No new deps, no FFI surface change.** SecretaryVaultAccess stays host-buildable via `swift test`; the app target needs the xcframework (built by `run-macos-tests.sh`).
- **Working-directory hazard bit twice this session** (cwd drifted main↔worktree between Bash calls). All commits verified on-branch; `main` untouched at `eac0d3df`. Resume commands below use explicit paths.
