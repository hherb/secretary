## (3) Open decisions and risks

### The big one: Task 11 (L4 e2e) was deferred, not implemented

**Decision (user, this session):** defer Task 11 and ship D.1.1 via Task 12. Recorded as issue **#161**.

**Why:** the plan's Task 11 assumed `tauri-driver` "shells out to `safaridriver` on macOS". **This is factually wrong.** Apple provides no WebDriver for WKWebView (the webview Tauri embeds on macOS); `safaridriver` automates Safari the browser, not an embedded WKWebView. The [official Tauri WebDriver docs](https://v2.tauri.app/develop/tests/webdriver/) support only Linux + Windows. So the planned harness cannot run on the macOS dev box. As of early 2026 the macOS options are community plugins (`tauri-plugin-automation` + `@crabnebula/tauri-driver` / `danielraffel/tauri-webdriver` / `Choochmeque/tauri-webdriver`) which add an unaudited plugin to the security-sensitive desktop app. L4 is explicitly **not a CI gate for D.1.1** (spec §13), so deferring does not block the ship.

**Consequence for whoever picks up #161:** choose between a Linux-targeted harness (officially supported, runs in future CI, doesn't run on the Mac) and the macOS plugin route (runs on the Mac, needs a deliberate dependency review). The PathPicker read-only-input workaround is filed separately as **#162** because it's needed regardless of which platform path the e2e takes.
