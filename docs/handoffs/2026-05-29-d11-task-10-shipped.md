### Plan execution trace (for the reviewer)

The plan as-written has several places where prior batons' drift carry-forwards bit; this session adapted:

1. **`App.svelte` already had the `vault-locked` event listener** (from earlier in D.1.1 — Task 5 or Task 6 carry-forward). The plan's `App.svelte` redo would have stomped existing code with a `sessionState.set({status:'locked', lastError:null})` raw mutation (which can't compile post-#150). Kept the existing listener; added only the activity-tracking `$effect` and the Toast rendering. Task 10 was narrower than the plan suggested in line count, but exactly as wide in contract surface.
2. **`$derived` shielding of the activity-tracking effect** was discovered red on the "does not double-start on settingsUpdated" test (start spy called 2× instead of 1×). The plan's body `$effect(() => { if ($sessionState.status !== 'unlocked') return; ... })` re-fires on every `$sessionState` change because the whole store is the dep. Wrapped in `const sessionStatus = $derived($sessionState.status)` so the effect dep dedupes via `===`. Documented in App.svelte as load-bearing for the contract.
3. **`<style>` block in Toast.svelte** would re-trigger the Vite 6 / Vitest `preprocessCSS` bug (#153 carryover from Task 8). All styles in `src/theme.css` per the project convention — appended a `.toast*` block (~55 LOC).
4. **Toast prop shape: discriminated-union `notice`, not a `message: string`** as the plan suggested. The plan's `message` prop forces App.svelte (or some intermediate) to compute the copy per reason; the actual auto_lock semantics live on the typed reason, so the Toast component owns the reason-to-copy mapping. The result is a switch that's exhaustive at compile time over the union (after the fixup-time union narrow).
5. **No `pnpm tauri dev` smoke this session.** Same gap as Tasks 7-9 — no UI environment in this session. ### Code-review fixups

`/code-review high` on PR #157 flagged 6 in-scope items (2 real correctness, 4 cleanup/altitude) + 2 pre-existing concerns in `auto_lock.ts` filed as separate issues per [[feedback_act_on_issues_dont_mention]]. All 6 in-PR items applied in one fixup commit (`8767bf6`) on top of the two code commits + this baton update:

1. **Filter altitude — push the `manual` filter into the producer.** `AutoLockNotice` no longer carries a `'manual'` variant; the union is now `{ reason: 'idle' } | { reason: 'keep_alive_failing' }`. `vaultLocked('manual', ...)` transitions state to `locked` but does NOT write the notice — the user clicked Lock themselves; no confirmation surface needed. Eliminated three coupled sites in one move: the union variant, the App `{#if … reason !== 'manual'}` filter expression, the Toast `case 'manual': return ''` dead arm + its defensive test. Result: one source of truth (the producer), exhaustive switch over a narrow union, dead arms can no longer accumulate.

2. **Clear stale `autoLockNotice` on `unlockSucceeded`.** Spec §12 ties the notice to the lock transition, not to the cross-session interval. Previously: a user who returned to an undismissed auto-lock toast (the 5s dismiss timer hadn't fired) and entered their password would see the stale notice linger on top of the freshly-mounted Vault for the remainder of the window. Fixed in `unlockSucceeded()` itself — one line: `autoLockNotice.set(null);` at the start. Two new tests pin "stale idle notice cleared on unlock" and "stale keep_alive_failing notice cleared on unlock".

3. **Toast `$effect` dep keyed on `notice.at`, not `void notice;`.** The new dep is semantically meaningful (a fresh notice = new `at` = fresh dismiss window, which is exactly what the existing "fresh notice resets timer" test pins) and resilient to refactors that might destructure the prop. `void notice.at;` makes the read explicit and reads the prop getter (whose access has side effects in Svelte 5), which esbuild cannot strip.

4. **App.svelte `$effect` returns `startActivityTracking()` directly** instead of wrapping it in `return () => { stop(); };`. The function's `() => void` signature already matches Svelte's cleanup contract; the wrapper was trivial indirection.

5. **`settingsUpdated` hoisted to the static import block in `App.test.ts`.** The previous mid-test `await import('../src/lib/stores')` was inconsistent with the file's six other static imports from the same module and hid a usage site from import-block grep.

6. **Tests updated for the narrowed union.** `stores.test.ts`: "from unlocked → locked via vaultLocked('manual')" now asserts `autoLockNotice` is null; added "vaultLocked('idle') raises notice" + "vaultLocked('manual') does NOT raise" + the unlockSucceeded-clears-stale describe block. `App.test.ts`: "reason='explicit' → autoLockNotice='manual'" inverted to "reason='explicit' does NOT raise"; "does NOT render Toast after explicit-lock" updated to assert notice is null (was: filtered). `Toast.test.ts`: removed `MANUAL_NOTICE` + defensive test (type-impossible).

## (3) Open decisions and risks

### Plan adaptations (worth flagging to the reviewer)

1. **App.svelte's existing `vault-locked` listener kept as-is.** The plan would have stomped it with a non-compiling `sessionState.set(...)`. The two patterns (onMount + manual-unmounted-flag for the listener; `$effect` + cleanup-return for activity tracking) co-exist — the listener pattern is async (`listen()` returns a Promise) and uses the closure-local `unmounted` flag for race-safe cleanup; the activity-tracking pattern is synchronous start/stop and uses the idiomatic `$effect` cleanup. Mixing is intentional, not accidental — they fit their respective requirements.
2. **`$derived` shielding for the activity-tracking effect.** Load-bearing for the "does not restart on settingsUpdated" contract. A future contributor who inlines `$sessionState.status` (thinking the `$derived` wrapper is incidental) would silently re-introduce the bug; the comment names this explicitly.
3. **Producer-side filter for the `manual` reason.** The biggest fixup decision — narrows the `AutoLockNotice` union, removes three coupled sites in one move. Worth flagging to the reviewer because it's a deliberate altitude choice over the plan's implied "filter at the render boundary".
4. **Toast prop is the discriminated-union `notice`, not a `message: string`.** Keeps reason semantics typed all the way to the rendering site; the switch is exhaustive at compile time.
5. **`unlockSucceeded` clears `autoLockNotice` as a producer-side guarantee.** Could have lived in App.svelte as an `$effect` on the unlocking → unlocked transition, but the cleaner altitude is "the new unlocked session starts fresh, owned by the same helper that creates it". Symmetric with the `vaultLocked` producer-side filter on `manual`.
6. **`auto_lock.ts`-side concerns (counter reset across cycles + race after manual lock) filed as separate issues** (#158 + #159), not fixed in this PR. Both are pre-existing in `auto_lock.ts`; this PR's surface (the Toast) makes them more reachable but the underlying gaps belong to `auto_lock.ts`'s contract. Fixing here would scope-creep the PR and split the auto_lock fix across two sites.

### Decisions settled

- **`AutoLockNotice` no longer has a `manual` variant.** The producer filter (in `vaultLocked`) is the only filter; downstream consumers see only reasons that warrant a notice. Future contributors cannot accidentally write a `'manual'` notice — the type rejects it.
- **Toast styles in `theme.css`.** #153 carryover. Toast's block is ~55 LOC under `.toast*`. When #153 resolves, this block (and the existing `.settings-dialog*` + `.locking-splash*` blocks) can migrate back into their component `<style>` blocks together.
- **The activity-tracking `$effect` runs once at mount + on every status change (with `$derived` dedupe).** Cleanup runs on every dep change AND on unmount, so document listeners never outlive the component tree. Pinned by "stops activity tracking on unmount when unlocked" test.
- **Toast's auto-dismiss uses `setTimeout` + clearTimeout via the `$effect` cleanup**, not a manual timeout-id ref + onDestroy. Simpler; the `$effect` cleanup is the right Svelte 5 idiom for a per-render timer.
- **No `Toast`-level filtering of the `manual` reason.** Type-narrow at the producer; the component's switch is exhaustive. Anything else is duplication and invites drift.
- **Toast `$effect` keyed on `notice.at`, not the prop reference.** Semantic dep; the test's "fresh notice resets timer" contract is exactly about a new `at` value getting a fresh window.
