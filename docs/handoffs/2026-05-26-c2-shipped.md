### Plan ↔ reality reconciliations

Three deliberate deviations from the plan, all driven by Rust ecosystem facts the plan didn't anticipate:

| Plan note | Reality | Resolution |
|---|---|---|
| Plan suggests "short-lived `tokio::process::Child`" for spawning the two daemons. | `cli/` has no tokio dep; adding one just for this test would be a 30+ transitive-crate hit + ~10 sec of additional CI build time. `std::process::Command::spawn` returns `std::process::Child` with a `.try_wait()` + `.id()` API that covers the scenario without any async machinery. | Use `std::process::Command::spawn`; poll-and-sleep on `child.try_wait()` for the SIGTERM grace period. |
| Plan implies `kill(pid, SIGTERM)` via the libc syscall. | The workspace lint `unsafe_code = "forbid"` (root `Cargo.toml`) blocks direct `libc::kill` calls. The alternatives are (a) add `nix` as a dev-dep with a `signal` feature flag — adds a focused crate but pulls in `bitflags`, `cfg-if`, `libc` indirectly + extra build time; or (b) shell out to `/bin/kill -TERM <pid>`. POSIX mandates the `kill` utility's existence + flag semantics on every Unix host we target, so the shell-out is portable and zero-dep. | Shell out via `std::process::Command::new("kill").arg("-TERM").arg(pid.to_string()).status()`. The test asserts the `kill` invocation itself succeeds before asserting the daemon's response, so a missing `/bin/kill` (which would be a broken Unix install, not a real failure mode) surfaces clearly. |
| Plan describes "mutate the vault from a third process" between the two daemons. | Mutating the vault means committing a new block which requires re-unlocking the vault, generating a new block file, signing with both Ed25519 + ML-DSA-65, and writing a new manifest — that's a sizeable orchestration just for a test. The simpler convergence scenario — both daemons see the same initial vault state and converge to the disk clock — is sufficient for the spec's "both instances must agree on what they saw" guarantee. | Two daemons, no third-party mutation; both fire one `run_one` cycle each (via `--poll-interval-secs=1`), state-save on SIGTERM, both state files compared for byte-identical `highest_vector_clock_seen`. The "with third-party mutation" scenario can be added in a future C.2.x slice or in C.4 (cross-device convergence conformance, which is explicitly scoped for that case). |

```bash

VAULT=$(mktemp -d) && STATE=$(mktemp -d)
cp -R core/tests/data/golden_vault_001/. "$VAULT/"
(
  echo "correct horse battery staple"

    --password-stdin --non-interactive \
    --debounce-ms 50 --poll-interval-secs 1 --ready-window-ms 200 \

DAEMON_PID=$!
sleep 3
kill -TERM $DAEMON_PID
```

## (3) Open decisions and risks

### Decisions settled during this session

- **`#![cfg(unix)]` on the entire `two_instance_convergence.rs` file** rather than per-test gates. The whole test depends on SIGTERM-driven shutdown, which the daemon installs only on Unix (per spec §D10 + `cli/src/signal.rs`'s `#[cfg(unix)]` blocks). Running this test on non-Unix would test "does SIGKILL on Windows produce a usable state file?" — which the spec explicitly does not promise (D10).
- **Shell out to `/bin/kill -TERM <pid>` for signal sending** rather than adding `nix` as a dev-dep. POSIX-guaranteed utility, zero new deps, side-steps `unsafe_code = "forbid"`. The test asserts the `kill` invocation succeeds before asserting the daemon's response, so a missing `/bin/kill` (broken Unix install) fails loudly rather than silently waiting forever.
- **`--poll-interval-secs=1` is the trigger mechanism** for the convergence test rather than mutating the vault folder to fire a notify event. `poll_interval` is `u64`-typed so 1 s is the floor (0 = disabled); using it gives a deterministic "both daemons sync within their first 1.2 s" guarantee, independent of cross-platform notify event-delivery latency. Total test wall-clock is ~3.5 s on macOS; comparable on Linux.
- **`/bin/kill -TERM` rather than `pkill` or `kill -SIGTERM`.** POSIX mandates `kill -<signal-number-or-name>` syntax; `-TERM` is the portable short form for SIGTERM (= 15). `pkill` is not POSIX-mandated and is absent on minimal containers.
- **Force-kill (SIGKILL) is the panic-safe fallback** in the convergence test. If a daemon doesn't exit within [`SIGTERM_GRACE`] (5 s), the test calls [`force_kill`] (which sends SIGKILL via `Child::kill`) before panicking. This prevents flaky CI runs from leaking long-running secretary-sync processes that would hold their lockfiles and break subsequent test runs.
- **Stdout/stderr piped (not inherited) on spawned daemons** so the test runner's diagnostic output doesn't get interleaved with the daemon's `tracing` events. The captured streams are drained implicitly when `Child::wait` returns.
- **`raw_notify_*` tests assert "at least one sync-relevant event arrives" rather than pinning a specific Create/Modify sequence.** Cross-platform `notify` taxonomy is fluid (Linux inotify emits Create-then-Modify; macOS FSEvents coalesces; both vary by file-creation syscall path). Pinning a specific sequence would make the test fragile against future kernel or `notify`-crate updates without adding signal — the regression we want to catch is "notify stops delivering events for writes", not "notify changed which subkind of EventKind it uses".
- **Three integration test files in `cli/tests/`** is the final shape: `main_validate.rs` (args-layer contract from Task 1), `once_integration.rs` (full `once` binary end-to-end from Task 9), `pipeline_integration.rs` (library API from Task 5 / 9), `two_instance_convergence.rs` (daemon end-to-end from Task 10), `notify_quirk.rs` (cross-platform notify pin from Task 10). Five files total, one concept each, all under the 500-LOC `feedback_split_files_proactively` threshold.

### Risks carried into C.3 / C.4 / D.1

- **`notify`-version pin discipline.** `cli/tests/notify_quirk.rs` exists specifically so a future `notify` major bump regression surfaces in CI. The current pin is `notify = "6"` in `cli/Cargo.toml`; if a contributor bumps to 7.x without re-reading this test's docstrings, they may misinterpret a real semantic change as a test bug. The fix is to read [`cli/tests/notify_quirk.rs`](../../cli/tests/notify_quirk.rs)'s top-of-file doc — the per-OS expectations there are the spec's contract with `notify`, not a quirk of one version.
- **`two_instance_convergence.rs` wall-clock budget.** Test takes ~3.5 s on macOS; loaded CI could push toward 5 s. The constants ([`SYNC_QUIESCENCE`] = 3 s, [`SIGTERM_GRACE`] = 5 s) have explicit doc-comments explaining the budget; if a CI runner times out, those are the knobs to bump (and the daemon's [`shutdown_poll_interval`] default of 1 s is the underlying lower bound). [#123](https://github.com/hherb/secretary/issues/123) is the parent issue tracking daemon-loop wall-clock test flakes.
- **The convergence test doesn't exercise concurrent mutation.** The spec §D8 acceptance criterion says "convergence across two instances"; this PR satisfies it for the read-only scenario. The mutation-while-watching scenario is C.4's job. If a contributor reads "convergence test" and assumes it covers concurrent writes, they will be wrong — this PR's docstrings explicitly call this out, but the test name itself doesn't.
- **`#![cfg(unix)]` means the convergence test is silently absent on Windows CI.** The test counts above (955 → 960) are for macOS and Linux; on a Windows runner the count would be 955 → 959 (the linux-gated notify test would also be absent, balanced by the macos-gated one being absent). We don't run Windows CI per `feedback_windows_not_primary`, but if we ever do, the test counts will diverge by one.
- **No Windows runner means signal-driven shutdown is untested cross-platform.** This is per `feedback_windows_not_primary` — the user considers Windows insecure-by-design and the project does not promise it. If a contributor wants to bring up a Windows port, the daemon's signal handling will need to be re-architected (Ctrl-C handler via `SetConsoleCtrlHandler` instead of signal-hook; the `ShutdownGuard` shape from `cli/src/signal.rs` would still apply).
