# Handoff — 2026-05-09 — Sub-project B.4b design spec

**Session type:** brainstorm + spec write (no code)
**Outcome:** B.4b design approved + committed to `main`. Implementation pending.

## Context entering the session

Per the post-merge baton in `NEXT_SESSION.md` from the 2026-05-06 B.4a session: PR #28 was squash-merged into `main` as `293e5c8`. The next step was to begin Sub-project B.4b (read individual encrypted blocks from an open vault), which required a brainstorm to resolve four open design questions before writing a spec or implementation plan:

1. Record type FFI projection — how do `Record` / `RecordField` / `RecordFieldValue` (with `SecretString`/`SecretBytes` zeroize-typed wrappers) project to Python / Swift / Kotlin while preserving zeroize discipline?
2. `read_block` API shape — consume `OpenVaultManifest` (one-shot) or take by reference (multi-block)? `Mutex<Option<T>>` enables both.
3. Error variant for block-not-found — add `FfiVaultError::BlockNotFound { uuid }` (7th variant) or fold into `CorruptVault`?
4. Trash entries — defer to Sub-project C (sync) or surface in B.4b with soft-delete flag on `BlockSummary`?

Workspace state at session start was the post-merge baseline (522 + 9 cargo, 30 pytest after applying the documented nuclear cache fix, 18/18 Swift, 19 Kotlin), all green.

## Decisions reached during the brainstorm

The session worked through the four primary questions one at a time using the brainstorming skill, then batched the smaller follow-up questions into a consolidated proposal. All decisions lock in cleanly with no carried-over ambiguity.

### Q1 — Record type FFI projection: **Hybrid**

`Record` is value-typed in **non-secret metadata** (record_uuid, record_type, tags, timestamps, tombstone) and opaque in **secret payload** (each `RecordField` becomes a `FieldHandle` with `expose_text()` / `expose_bytes()` accessors). The "bytes-not-strings" discipline from B.2/B.3a/B.3b doesn't fit cleanly here because `RecordFieldValue::Text` is genuinely human-readable text — but the Hybrid resolves this by making the secret-pull the explicit, opt-in step on a per-field basis.

Rejected alternatives:
- *Eager copy-out value type* — projects `Record` as a foreign dataclass / struct / data class. Simplest API; weakest hygiene (secret bytes linger in foreign-language strings/bytes until GC).
- *Opaque Record + accessor functions* — strongest hygiene; busiest API.

### Q2 — `read_block` API shape: **Free function on the bridge**

`bridge::read_block(&UnlockedIdentity, &OpenVaultManifest, &[u8; 16]) -> Result<BlockReadOutput, FfiVaultError>`. Free-function shape matches the user's "pure functions in reusable modules over complex code" preference (project memory `feedback_pure_functions`).

Important refinement during exploration: the original NEXT_SESSION question framed this as "consume vs reference" but missed that `read_block` needs the **asymmetric secret keys (X25519 + ML-KEM-768)** to perform hybrid-decap on the per-block KEM wrap — those live in `IdentityBundle` inside `UnlockedIdentity`, NOT in `OpenVaultManifest` (which only holds the symmetric `identity_block_key` for manifest-body decryption). So `read_block` is fundamentally a two-handle operation.

Rejected alternatives:
- *Method on `OpenVaultManifest`* — vault-centric reading; foreign caller still has to pass identity as argument.
- *Method on `UnlockedIdentity`* — conceptually backwards; the focus is the vault, not the identity.

### Q3 — Error variant for block-not-found: **+1 variant `BlockNotFound`**

`FfiVaultError` grows from 6 → 7 variants. New `BlockNotFound { uuid_hex: String }` distinguishes "you asked for a block that doesn't exist" (caller can refresh / re-check) from "the vault is broken" (`CorruptVault`). File-missing folds into `CorruptVault` (manifest+disk diverged = vault integrity issue); decrypt failures (`NotARecipient`, signature fail, AAD mismatch, `BlockUuidMismatch`, `AuthorFingerprintMismatch`) all fold into `CorruptVault` per the same anti-conflation discipline established for `FfiUnlockError`'s `WrongPasswordOrCorrupt`.

Rejected alternatives:
- *+2 variants (BlockNotFound + BlockFileMissing)* — surfaces sync-race transients distinctly but B.4b doesn't model sync.
- *Option<Record list> + fold file-missing into CorruptVault* — conflates "caller asked for nonexistent block" (mistake) with success-with-nothing.

### Q4 — Trash entries: **Defer to Sub-project C**

`block_summaries()` continues to filter out trashed blocks. `read_block(trashed_uuid)` returns `BlockNotFound`. No `is_trashed: bool` flag, no `tombstoned_at_ms` in `BlockSummary`. Sub-project C's sync orchestrator will add the restore-from-trash flow with full vector-clock context — surfacing trash here without that context would be incomplete (resurrection conflict detection requires sync-orchestrator knowledge).

### Follow-up decisions (batched)

- `Record.unknown` / `RecordField.unknown` (forward-compat CBOR) — **not surfaced**. Same rationale as B.4a's no-`BlockSummary.unknown` decision.
- `Record.tombstone: bool` — **surfaced** (UI affordance). `Record.tombstoned_at_ms` — **not surfaced** (CRDT internal).
- `Record.tags`, `record_type`, `record_uuid`, timestamps — surfaced as plain value-type fields.
- `RecordField.device_uuid` — surfaced as `bytes` / `Data` / `ByteArray` mirroring B.4a's `BlockSummary.recipient_uuids` projection.
- Output container — opaque `BlockReadOutput` handle with single cascading `wipe()`.
- `BlockPlaintext.schema_version` / `block_version` / `vector_clock` — **not surfaced** (v1 always 1; sync internal).
- Decrypt failure variants — all fold into `CorruptVault { detail }`.
- Conformance.py extension — **skip**; same rationale as B.4a (FFI surface is not part of the spec contract).
- New on-disk fixture — **none**; `golden_vault_001/blocks/11223344-...` already provides the read-block KAT.

### Section 3 follow-up: vault folder PathBuf placement

`OpenVaultManifestInner` is extended with `vault_folder: PathBuf` (bridge-internal change; no public B.4a accessor surface change). `open_vault_with_password` and `open_vault_with_recovery` already receive the folder path. B.4c (`save_block`) and B.4d (`share_block`) reuse this for their atomic-write paths.

Rejected alternative: passing `vault_folder: &str` as a third argument to `read_block` — keeps B.4a's manifest internals frozen but adds friction for the foreign caller.

### Section 5 follow-up: wrong-length UUID handling

PyO3 / uniffi wrappers raise `ValueError` / `IllegalArgumentException` on wrong-length UUID input (the bridge's `read_block` takes `&[u8; 16]` — compile-time enforced; only the FFI wrappers ever see wrong-length input). Folding into `BlockNotFound` was rejected because it would silently mask programmer bugs in foreign-language UUID-formatting code as "block not found" data errors.

### User-prompted hardening: fuzz target on `RecordFieldValue::Text` UTF-8

After the initial spec was committed, the user asked whether `expose_text()` invalid UTF-8 should be a fuzz target or surface as another error code. The structural guarantee was already in place (CBOR `tstr` requires valid UTF-8 per RFC 8949 §3.1; `ciborium` enforces this on decode; `parse_record_field` only constructs `RecordFieldValue::Text(SecretString::new(s))` from already-validated `Value::Text(s)`), so invalid UTF-8 cannot reach `expose_text()` by construction.

The right action is to extend the existing `core/fuzz/fuzz_targets/record.rs` target with a defense-in-depth assertion: after successful `record::decode`, walk every `RecordFieldValue::Text` and assert `s.expose()` is valid UTF-8. The assertion serves as a tripwire if the decode path is ever weakened to allow direct `SecretString` construction from non-validated bytes. Promoted from "open question" to "in scope for B.4b implementation" in the spec's Goals section §7 + Open questions §1.

The rejected alternative — surfacing a 7th-or-8th `FfiVaultError` variant via changing `expose_text() -> Option<String>` to `Result<Option<String>, FfiVaultError>` — would propagate a corruption-time concern (the block file was tampered with on disk) to a foreign-language access-time path (the caller is just reading a field), forcing every caller to handle an error that is structurally impossible in the v1 type contract. Worse, it would split the corruption-class signal between `read_block`'s return (which already maps decode failures to `CorruptVault`) and the per-field accessor — fragmenting the anti-conflation discipline established for `FfiVaultError`.
