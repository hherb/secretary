### Architecture (where the pieces live)

- `secret_source.rs` ‚Äî `DeviceSecretSource` trait (the port); `DevFileSecretSource`
  reads the 32-byte secret as hex from a cleartext file. **Dev-only**, loudly
  documented; real OS-keystore adapters land behind the same port later.
- `config.rs` ‚Äî `HostConfig { vault_path, device_uuid (hex), secret_source }`;
  `load_default()` returns `Ok(None)` for an absent config (the "not enrolled"
  state). `SecretSourceConfig` is `type`-tagged so keystore variants slot in.
- `vault.rs::per_fill_count` ‚Äî the per-fill open: fetch secret ‚Üí `open_vault`
  (the *same* verify-before-decrypt) ‚Üí `manifest.blocks.len()` ‚Üí drop
  secret/identity (zeroizing).
- `lib.rs::Context` + `run()` ‚Äî holds the source *port*, not the secret. `query`
  ‚Üí `per_fill_count` ‚Üí `available{count}`; `Secret(Unavailable)` ‚Üí `count:0`;
  other failure ‚Üí `error`.
- `enroll.rs` + `src/bin/enroll.rs` ‚Äî dev enrollment over the genuine
  `add_device_slot`.

## (3) Security invariants demonstrated (design ¬ß12)

- **1 (no key material between fills)** ‚Äî `Context` holds the *source*, not the
  secret; the secret is a `SecretBytes` fetched per open and dropped right after,
  and the `OpenVault` (IBK + identity) drops before the count returns.
- **2 (casual vault only)** ‚Äî the host opens only the configured vault; the
  high-value vault is never enrolled.
- **6 (one credential / no list)** ‚Äî forward-held trivially: only an integer
  `count` crosses.
- **7 (same verify-before-decrypt)** ‚Äî the open is literally
  `open_vault(Unlocker::DeviceSecret { .. })`, no parallel/weaker path.
- **Structural (D.4.1)** ‚Äî no socket, manifest-bound ID, never-panic 1 MiB
  framing all still hold.

## (4) What D.4.3 picks up

D.4.3 is **the security-critical slice**. It replaces the trivial
"all live blocks" count with the **real origin-matching engine** ‚Äî PSL (eTLD+1)
vs `exact_origin`, top-frame-governs, cross-origin-iframe refusal, HTTPS-only,
IDN de-confusion ‚Äî gated by a pinned `origin_match_kat.json` corpus (design ¬ß5),
still returning only a count (injection is D.4.4). It also lands the **real
OS-keystore `DeviceSecretSource` adapters** (behind the D.4.2 port) and the
**desktop enrollment UI** (replacing `secretary-browser-enroll` + the dev file
secret). Read design ¬ß5 first; it carries the most review.

## (5) Guardrails that must keep holding

- The per-fill open MUST stay `open_vault(Unlocker::DeviceSecret { .. })` ‚Äî the
  same verify-before-decrypt. No parallel/weaker open.
- No long-lived key material in the host; open per fill, drop immediately.
- No secrets cross the channel until D.4.4's native-confirmation gate.
- The `DevFileSecretSource` is dev-only ‚Äî do not let it become the production
  path; the real keystore adapters go behind the same port.
- `#![forbid(unsafe_code)]` + clippy `-D warnings` + never-panic 1 MiB framing.
