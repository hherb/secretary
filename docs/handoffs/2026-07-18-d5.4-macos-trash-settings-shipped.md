- **Code-review fixups** (post-review commit) — four findings from the PR review, all view-layer:
  1. **Settings numeric inputs are now text-buffered, committed explicitly at Save.** `TextField(value:format:)` commits its binding only on Return or focus loss, and an AppKit button click does not move first responder — so a typed-then-**mouse-clicked** Save could persist the *previous* value while the field still showed the new one, breaking the WYSIWYG contract `SettingsViewModel.save()` documents. It also silently saved the old value when a field was cleared (a failed parse leaves the bound value untouched). `MacSettingsView` now holds raw text, parses + pushes through the VM's clamping setters in `commitEdits()`, refuses the save on unparseable input, and re-seeds the fields from the clamped VM values so display == what is written. A `@FocusState`-defocus fix was considered and rejected: the resignation flushes on the next view update, so it races the `Task`. **No VM change** — the clamping/gate/retarget logic is untouched.
  2. **Trash sidebar refresh moved to `.sheet(item:onDismiss:)`.** It previously hung off the child's Done closure, so it fired on exactly one dismissal path; `onDismiss:` (the idiom the move picker already used) makes it path-independent.
  3. **Latent lock/sheet interaction documented.** `lock()` nils `blockNameDialog`/`movingRecord` but cannot see the three view-local `@State` sheets; they survive a lock and are torn down only because `onLock` flips the root route. Safe today (macOS blocks the parent's close button behind a modal sheet, and Lock sits behind the sheet) — commented so an in-place-lock refactor doesn't silently leave a sheet over a wiped session. Also noted that the `willClose` **window-identity filter is now load-bearing for sheets**: each sheet is its own `NSWindow` posting its own `willClose`, so without the guard, closing the Trash sheet would wipe the session.
  4. **`WindowAccessor` extracted** to `MacWindowAccessor.swift` — a general window utility had landed in the field-detail file during the D.5.4 extract.

  Also fixed during self-review of (1): the bad-input path returns before `viewModel.save()`, which is what clears the VM's `banner` — so a bad-input Save after a good one rendered "Settings saved" directly above "not saved". `banner` is `private(set)`, so the view now suppresses the VM's banner/error whenever `inputError` is set (it describes the newer attempt).

Toolbar order is now (six primary items, user-approved): Show-deleted · Add record · New block · **Trash** · **Settings** · Lock. Every write still funnels through the VM's existing write-reauth gate — the views never bypass it; same verify-before-decrypt / re-auth-before-write invariants as iOS/D.5.3.

**⚠️ Manual GUI smoke = the one outstanding pre-merge gate (author-run).** Same manual-only posture as D.5.3 / #161. The full loop to exercise from a signed Xcode run:
- **Settings**: toolbar → Settings; edit retention days + grace minutes; Save → write-reauth gate (Touch ID/password) prompts; reopen → values round-trip; "Settings saved" banner.
  - **Commit-path specific (review finding 1 — do NOT skip):** type a new value and click **Save with the mouse, without pressing Tab or Return first**, then reopen and confirm the *typed* value round-tripped. Pressing Return commits the field independently and would mask the bug this fixup addresses.
  - **Bad input:** clear a field entirely (or type letters) → Save must refuse with "Both fields need a whole number", write nothing, and must NOT show a stale "Settings saved" banner alongside it.
  - **Clamping:** type a value above the documented range → Save re-seeds the field to the clamped value that was actually written.
- **Trash**: toolbar → Trash; right-click a trashed block → Restore (gate prompts; block reappears in the sidebar after dismiss) and Delete-forever (confirm → gate); Empty trash (confirm → gate); Run retention now (preview → Purge N, or "nothing expired"). Confirm closing the nested retention sheet does NOT dismiss the Trash sheet.
- **Toolbar:** with six `.primaryAction` items now, narrow the window and confirm the overflow (») behaviour is sane and Lock stays reachable.
- **Seeding caveat:** there is currently **no in-app action to trash a *block*** on either iOS or macOS (the Trash view manages blocks trashed via sync or the core API). To smoke restore/purge, pre-seed a trashed block into a **temp copy** of the demo/golden vault (never mutate the tracked fixture — [[feedback_smoke_test_temp_copy_golden_vault]]). ## (3) Open decisions and risks

- **Scope was deliberately tight.** The device-slot toggle is out of D.5.4 by the user's choice — do not "finish" it into this PR. It is D.5.5.
- **The manual GUI smoke is the one outstanding pre-merge gate** (see §1). Everything else (host suites + app compile) is green and automated. Mirrors the D.5.3/D.1.14/D.1.15 posture ([#161](https://github.com/hherb/secretary/issues/161)-style).
- **Trash-view seeding.** No in-app block-trash action exists (pre-existing on iOS too), so the Trash view is only reachable-with-content via sync or a hand-seeded trashed block. This is faithful iOS parity, not a D.5.4 regression — but it means the restore/purge smoke needs a seeded block.
- **macOS interaction diverges from iOS by design.** iOS uses swipe actions + push-navigation; macOS uses context menus + toolbar-button → modal sheets (user-approved). Intentional — swipe/push are wrong idioms with a mouse and a split view.
- **VM built at tap, not per-render.** Both `makeTrashViewModel()` / `makeSettingsViewModel()` do an FFI read in the VM `init`, so the toolbar buttons build the VM on tap (guarded by `if let`) rather than gating button visibility on the factory (which would pay the FFI every render). On macOS the session always conforms to both ports, so the buttons always show.
- **`MacBrowseView+Fields` → `MacFieldDetailView`.** The extraction is a component, not an extension (Swift `private` is file-scoped). If a future reviewer expected the literal `+Fields.swift` filename, this is why.
- **No new deps, no FFI surface change.** `SecretaryVaultAccess` stays host-buildable via `swift test`; the app target needs the xcframework (built by `run-macos-tests.sh`).
