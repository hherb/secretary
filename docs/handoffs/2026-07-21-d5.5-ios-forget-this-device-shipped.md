### The Swift type-checker gotcha (read before touching `VaultBrowseScreen`)

Adding the single `.navigationDestination(item:)` modifier tipped the already-large `VaultBrowseScreen.body` over the compiler's limit ‚Äî **"unable to type-check this expression in reasonable time"** (exit 65, not a logic error). Fixed by extracting `browseToolbar` (`@ToolbarContentBuilder`) and `listContent` (`@ViewBuilder`) out of `body`. Both are behaviorally identical to the inline forms (verified against `dd1a996`). Also: `.navigationDestination(item:)` requires the item type be **`Hashable`** ‚Äî `SettingsRoute` holds view-model reference types, so it hashes/compares by its per-tap `id: UUID` (identity equality is exact because each route is minted fresh at tap). If you add more to this body, expect to keep it decomposed.

### The security ordering (carried across from macOS verbatim ‚Äî do not "simplify")

`DeviceSlotViewModel` owns it; the view only honours `.forgotten ‚Üí lock`.
- **Re-auth before revocation** (VM-enforced): the gate proves presence via the *same* enclave key the revocation deletes, and `GraceWindowReauthGate.authorizeWrite` short-circuits on `guard authorizer.isEnrolled`.
- **`.forgotten` means "this device's credential is gone ‚Üí lock", NOT "the revocation succeeded"** ‚Äî reached on full success AND on a partial teardown that already killed the enclave key (else the session runs with every write silently ungated). The `¬¨G ‚üπ ¬¨D` discriminator lives in the VM; the iOS `Task { await forget(); if state == .forgotten { onForgotten() } }` at `SettingsScreen.swift` mirrors macOS.
- **iOS-specific: revocation completes before `scoped.end()`.** `forget()` writes into the security-scoped vault folder (`remove_device_slot` ‚Üí `remove_file devices/<uuid>.wrap`); `onForgotten` fires only after `await forget()` returns, and `onLock` calls `scoped.end()` last. **"Dismiss then lock"** collapses to the route change: `route = .select` unmounts the pushed `SettingsScreen` ‚Äî no explicit `dismiss()` (would double-animate).

## (3) Open decisions and risks

- **iOS app views have no automated render coverage (#417).** Structural, unchanged here. Mitigation: all security-relevant logic lives in the host-tested `DeviceSlotViewModel`, not the view. The one seam that is manual-proof-only is `.forgotten ‚Üí onForgotten() ‚Üí lock` at the *view* layer (VM half is tested) ‚Äî step 3 of the smoke.
- **`DeviceUnlockScreen.swift` kept** (user decision) ‚Äî dead but retained as a raw-`LAError` debug harness; still referenced only in its own declaration + one `ios/README.md` line.
- **Fresh-auth-on-destructive is still an unfiled cross-platform gap** (carried from the D.5.5 baton, item 2). The grace window (default 2 min) lets a recent unrelated write satisfy `authorizeWrite`, so Forget can run with no biometric prompt within the window. Worst case is a *denial* (attacker at a recently-unlocked device revokes enrollment; user recovers with the master password), and the destructive-role confirmation dialog still fires. **Do not re-file** ‚Äî known and consciously unfiled; the right fix is a `requireFreshAuth` parameter on `WriteReauthGate.authorizeWrite` as its own slice.
- **Type-checker fragility.** `VaultBrowseScreen.body` sits near the Swift inference limit; the extractions bought headroom but further additions will need the same decomposition.
