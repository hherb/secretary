## (1) What we shipped this session

**The central idea:** the two one-shot secret accessors on the uniffi surface — `MnemonicOutput.take_phrase()` and `DeviceSecretOutput.take_secret()` — were declared `sequence<u8>?` in the UDL, which uniffi materializes as a **boxed `List<UByte>?`** on Kotlin (and `[UInt8]?` on Swift). The Kotlin boxed list **cannot be zeroized in place**, so during device-slot enrollment (and recovery-phrase display) the raw 32-byte secret / 24-word phrase lingered in a non-overwritable heap object until GC. Declaring both accessors **`bytes?`** makes uniffi hand back a zeroizable **`ByteArray?` (Kotlin) / `Data?` (Swift)** instead.

**Why the Rust side didn't change:** uniffi maps *both* `sequence<u8>` and `bytes` to Rust `Vec<u8>`, so the bridge impls (`Option<Vec<u8>>` in `ffi/secretary-ffi-bridge/src/{create,device}.rs` + the uniffi wrappers) are byte-identical. Only the **foreign container type** flips. Hence **no conformance-KAT byte-format impact** (the issue's main risk, now ruled out).

| **UDL** | `ffi/secretary-ffi-uniffi/src/secretary.udl`: `take_phrase()` + `take_secret()` → `bytes?` (was `sequence<u8>?`); doc comments note the zeroizable-bytes guarantee + #261. |
| **Android consumer** | `android/kit/.../UniffiVaultDeviceSlotPort.kt`: `takeSecret()` is now `ByteArray?` directly — dropped the `ByteArray(taken.size){…}` boxed-list copy. |

### Verification of the binding type flip (so a future reader trusts it)
After the UDL edit, regenerating the bindings shows the foreign type changed, NOT the Rust side:
- Swift: `func takeSecret() -> Data?` / `func takePhrase() -> Data?` (was `[UInt8]?`).
- Kotlin: `fun takeSecret(): kotlin.ByteArray?` / `fun takePhrase(): kotlin.ByteArray?` (was `List<UByte>?`).
- The committed `ffi/secretary-ffi-uniffi/bindings/{swift,kotlin}/` are **gitignored build artifacts** — regenerated by the run.sh / run_conformance.sh pipelines and by the android `:kit` gradle build + the iOS `build-xcframework.sh`. Nothing under `bindings/` is committed; do not hand-edit it.

### Deliberate design decisions (so a future reader doesn't "fix" them)
- **Rust bridge signatures stay `Option<Vec<u8>>`.** Don't "match" the UDL by changing them to a bytes-specific type — uniffi already maps `bytes` ↔ `Vec<u8>`. Changing them buys nothing and risks the scaffolding/impl mismatch.
- **Both accessors changed, not just `take_secret()`.** The issue flagged `take_secret()` but named `take_phrase()` as the sibling to check; a half-fix would leave the 24-word recovery phrase still boxed. Both now return `bytes?`.
- **iOS keeps the `[UInt8]` port contract.** `EnrolledSlot.deviceSecret` / `CreatedVault.phrase` are `[UInt8]` (already zeroizable Swift arrays — never the boxed-list problem). The consumers convert `Data → [UInt8]` and zero the transient `Data`, rather than widening the port types to `Data`. This keeps the blast radius to two files and leaves the pure `SecretaryDeviceUnlock` / `SecretaryVaultAccess` packages untouched.
- **No README change.** README never mentioned the residue; adding a note would be minutiae (per the brief-README convention). The ROADMAP row is the record.

## (3) Open decisions and risks
- **No on-device run was needed.** This is a representation-only change proven by the conformance harnesses (which exercise enrol_round_trip + the device-secret vectors through the *real* generated bindings on both languages) + the full iOS XCTest suite + the android `:kit` host compile. No new crypto, no new on-disk bytes, no behavior change — only the heap container the secret briefly lives in. A device smoke would add no signal the harnesses don't already give.
- **The transient `Data` on iOS is zeroized but `Data`'s internal copy-on-write backing is not guaranteed contiguous-and-unique** — `resetBytes` zeros the current buffer, which is the conservative best-effort already used elsewhere; the canonical secret remains the `[UInt8]` copy the coordinator/view-model zeroizes. This is an improvement over the prior code (which had no `Data` intermediate at all, so nothing regressed), not a new guarantee.
