### The security bug the final review caught — read this before touching the lock path

The feature rests on an ordering constraint: the write-reauth gate proves presence by driving the **same per-vault Secure-Enclave key that the revocation deletes**, and `GraceWindowReauthGate.authorizeWrite` opens with `guard authorizer.isEnrolled else { return }`. So re-auth must precede the revocation, and the session must lock after it — otherwise the rest of the session runs with every write silently ungated.

The per-task reviews all passed. The **whole-branch review found a path none of them could see**: `disenroll` is **not atomic**. `SecureEnclaveDeviceSecretStore.clear()` deliberately attempts **both** `SecItemDelete`s (SE key and wrapped blob) before throwing the first hard failure, and the gate's predicate is *blob presence alone*. So the blob delete can land while the key delete fails → `disenroll` throws → the VM stayed `.idle` → the view never locked — **yet the gate was already dead**. Exactly the outcome the design existed to prevent, reached sideways.

Fixed across four commits (`cacb080` security, `0376f29` error copy, `8eed5e6` symmetric disable, `77d032a` no-op test) plus `65fd950` (comment corrections):

- **`cacb080`** re-reads `port.isEnrolled` in the revocation catch arms and sets `.forgotten` (which the view locks on) when the credential is gone. Deliberately **not** applied to the `authorizeWrite` catch arm — a cancelled re-auth precedes all teardown.
- **Why the discriminator is sound** (an implication, not a case analysis): gate `G = enclave.isEnrolled` (blob alone); port `D = enclave.isEnrolled && metadata != nil`. So `D ⟹ G`, hence **`¬G ⟹ ¬D`** — the gate can never be dead while `port.isEnrolled` still reads true. No false negative in the dangerous direction, structurally. The converse is deliberately loose: a transient metadata read error also reads false and **over-locks an intact session**, which is fail-safe (costs a master-password re-entry).
- **`.forgotten` now means "this device's credential is gone, so lock" — NOT "the revocation succeeded."** `65fd950` corrected the `MacSettingsView` call-site comment, which still asserted the negation of the fix; left as-is, a contributor tidying for consistency would have read it as the invariant and reverted the fix.

Both reviewers verified by **mutation, not reading**: the Task 2 reviewer transposed the gate/revocation calls and deleted the `isBusy` guard (both correctly failed); the fix reviewer reverted the security fix both ways — removing the guard fails exactly `testPartialFailureThatTearsDownCredentialStillLocks`, making it unconditional fails `testPortFailureDoesNotReachForgotten` + `testRetryAfterFailureSucceeds`. The two tests genuinely bracket the discriminator in both directions.

## (3) Open decisions and risks

- **Three follow-up issues drafted, awaiting your OK to file** (`.superpowers/sdd/progress.md` holds the full text):
  1. **iOS parity** — the "Forget this device" section in iOS Settings. All logic already exists and is host-tested; this is view wiring only. Draft body at `docs/superpowers/plans/2026-07-19-d5.5-followup-ios-issue.md`.
  2. **Fresh-auth-on-destructive (cross-platform).** The v1 grace window (default 2 min) lets a recent unrelated write satisfy `authorizeWrite`, so an irreversible one-shot action can run with **no biometric prompt**. Both the final reviewer and I judged this **not a D.5.5 blocker** and explicitly **not** something to special-case in `DeviceSlotViewModel`: the same shape applies to empty-trash, delete-forever and run-retention-now on macOS + iOS + Android + desktop. The right fix is a `requireFreshAuth` parameter on `WriteReauthGate.authorizeWrite` adopted by every irreversible action — its own slice. Worst case here is a *denial*, not a disclosure: an attacker at a recently-used unlocked Mac can revoke biometric enrollment; the user recovers with the master password and re-enrols. The destructive-role confirmation dialog still fires, so the action is never silent — only the biometric check can be skipped.
  3. **`SecureEnclaveDeviceSecretStore.clear()` partial-failure recovery (PRE-EXISTING, not introduced here).** If the blob delete fails while the key delete succeeds, the device reads as enrolled but `authorize()` can only ever throw `.notEnrolled` — so Forget can no longer be completed from the UI, because the gate throws before the revocation is reached. Fail-closed, unreachable without a Keychain fault, no recovery short of relaunch.
- **The macOS view layer still has no automated render coverage.** Structural and unchanged by this slice (`run-macos-tests.sh` compile-proves the app but runs no test action against it). Mitigation: all security-relevant logic lives in the host-tested VM, not the view. The one seam that is manual-proof-only is `state == .forgotten → dismiss → onLock()`; the VM half is tested, that the *view* honours it is step 3 of the smoke.
- **`CoordinatorDeviceSlotPort` has no test at any level.** `SecretaryKit` is iOS/macOS-only and links the xcframework, so it cannot host-build; it is compile-proven only. The behaviour it adapts is covered by `DeviceUnlockCoordinatorTests`, but the error *mapping* is unpinned.
- **Scope was deliberately tight.** Enroll-from-Settings is still deferred. Managing *other* devices' slots is **not expressible at any layer** — adding it means new core + bridge + udl surface, not a UI slice.
- **Worktree hazard hit this session.** The Bash cwd silently reverted to the main checkout mid-session; a `git log` there showed `3c8f083` and a stale `.superpowers/sdd/` from an unrelated July 7 session, briefly looking like lost work. Nothing was lost. Every subagent dispatch after that carried an explicit `cd <worktree> &&` instruction. Keep doing that ([[feedback_edit_tool_targets_main_not_worktree]]).
