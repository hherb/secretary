**Title:** iOS Settings: add the "Forget this device" section over the shared DeviceSlotViewModel (macOS shipped in D.5.5)

**Labels:** ios, enhancement

## Context

D.5.5 added a "This Mac" section to the macOS Settings screen that revokes this
device's per-device wrap slot (ADR 0009), clears the Secure-Enclave key and
enrollment metadata, and locks the session.

All of the logic landed in the SHARED, host-tested packages:

- `DeviceSlotPort` ‚Äî `ios/SecretaryVaultAccess/Sources/SecretaryVaultAccess/DeviceSlotPort.swift`
- `DeviceSlotViewModel` ‚Äî `ios/SecretaryVaultAccess/Sources/SecretaryVaultAccessUI/DeviceSlotViewModel.swift`
- `VaultBrowseViewModel.makeDeviceSlotViewModel(port:)`
- `CoordinatorDeviceSlotPort` ‚Äî `ios/SecretaryKit/Sources/SecretaryKit/DeviceUnlock/CoordinatorDeviceSlotPort.swift`

The view model is deliberately platform-neutral (its re-auth reason says "this
device", not "this Mac"), so iOS needs only the view wiring.

## The gap

iOS device enrollment is one-way today: the
`Toggle("Remember this device with Face ID")` at `ios/SecretaryApp/Sources/UnlockScreen.swift:68`
renders only when `!biometricEnrolled`, so once enrolled it disappears and can
never be switched off. `ios/SecretaryApp/Sources/SettingsScreen.swift` has no
device section at all. The only `disenroll()` call site in iOS app code is
`ios/SecretaryApp/Sources/DeviceUnlockScreen.swift:33`, which is dead code ‚Äî
`DeviceUnlockScreen` is referenced nowhere outside its own declaration.

## Scope

- Add a "This Device" `Section` to `SettingsScreen` mirroring `MacSettingsView`'s,
  with a `.destructive` button + confirmation, hidden when not enrolled.
- Build the port in `SecretaryApp.swift` where `VaultBrowseViewModel` is
  constructed (it has the vault path) and thread it to the settings route.
- On `.forgotten`: dismiss, then lock ‚Äî for the reason below.
- Consider deleting the dead `DeviceUnlockScreen.swift` in the same PR.

## Care required ‚Äî the ordering constraint

Carry the macOS reasoning across verbatim. The write-reauth gate proves presence
via the SAME per-vault Secure-Enclave key that `disenroll()` clears, and
`GraceWindowReauthGate.authorizeWrite` opens with
`guard authorizer.isEnrolled else { return }`. So:

- Re-auth must be requested BEFORE the revocation (the VM already guarantees this).
- The session MUST lock after a successful revocation, or the rest of the session
  runs with every write silently ungated.

The VM enforces both halves; the iOS view must honour the `.forgotten` ‚Üí lock
contract exactly as `MacSettingsView` does.

## Acceptance

- Section present in iOS Settings when enrolled, absent when not.
- Face ID prompt on Forget; session locks back to the unlock screen.
- After forgetting, the unlock screen offers the password path with the "Remember
  this device" toggle restored.
- Cancel path leaves the device still enrolled.
- Existing host suites stay green; no new view model or port is needed.
- On-device proof on real hardware (Face ID cannot be exercised by any automated gate).

## Non-goals

- Enroll-from-Settings (needs a master-password re-prompt; deferred on both platforms).
- Managing other devices' slots ‚Äî there is no slot-enumeration API at any layer.
