# Design: iOS "Forget this device" Settings section (#461)

**Date:** 2026-07-21
**Issue:** [#461](https://github.com/hherb/secretary/issues/461) — iOS parity for the macOS D.5.5 "Forget this Mac" device-slot revocation.
**Branch:** `feature/d5.5-ios-forget-this-device` (worktree `.worktrees/d5.5-ios-forget-this-device`).

## Overview

D.5.5 (PR #460) added a "This Mac" section to the **macOS** Settings screen that
revokes this device's per-device wrap slot (ADR 0009), clears the Secure-Enclave
key + enrollment metadata, and locks the session. All the security-critical
logic landed in the **shared, host-tested** packages:

- `DeviceSlotPort` / `FakeDeviceSlotPort` — `ios/SecretaryVaultAccess/.../DeviceSlotPort.swift`
- `DeviceSlotViewModel` — `ios/SecretaryVaultAccess/.../SecretaryVaultAccessUI/DeviceSlotViewModel.swift`
- `VaultBrowseViewModel.makeDeviceSlotViewModel(port:)`
- `CoordinatorDeviceSlotPort` — `ios/SecretaryKit/.../DeviceUnlock/CoordinatorDeviceSlotPort.swift`
- `deviceSlotErrorMessage(_:)` — `ios/SecretaryVaultAccess/.../SecretaryVaultAccessUI/SettingsErrorMessage.swift`

The view model is deliberately platform-neutral (its re-auth reason says "this
device", not "this Mac"), so **iOS needs only the view wiring**. This slice is UI
plus orchestration — **no new cryptography, no new FFI surface, no new view model
or port, zero `core` / `.udl` / `FfiVaultError` / bridge / on-disk-format change.**

## The gap being closed

iOS device enrollment is one-way today:
`Toggle("Remember this device with Face ID")` at
[UnlockScreen.swift:68](../../../ios/SecretaryApp/Sources/UnlockScreen.swift#L68)
renders only when `!biometricEnrolled`, so once enrolled it disappears and can
never be switched off. `SettingsScreen.swift` has no device section at all. The
only `disenroll()` call site in iOS app code is inside `DeviceUnlockScreen.swift`,
which is unreachable dead code (a walking-skeleton harness — **kept**, see
Non-goals).

## Goals

- Add a "This Device" `Section` to the iOS `SettingsScreen`, mirroring
  `MacSettingsView`'s "This Mac" section: a `.destructive` "Forget This Device"
  button + confirmation dialog, hidden when not enrolled.
- Build the `DeviceSlotPort` where the vault path lives (`RootView`, the iOS
  analogue of `MacRootView`) and thread it to the settings route.
- Honour the `.forgotten` → lock contract exactly as `MacSettingsView` does.

## Non-goals (deliberate)

- **Keep** `DeviceUnlockScreen.swift` — it surfaces raw `LAError` domain+code that
  the product UI does not, a useful on-device biometric debugging harness.
- **#459** (iOS `TextField(value:format:)` stale-save) — the numeric fields stay
  as-is; it is a separate tracked issue needing on-device repro first. The forget
  section does not touch those fields.
- ~~**No manual "Lock" toolbar button**~~ — deferred at design time as scope
  creep, but **added post-review** (`e70f8cd`) after the on-device smoke surfaced
  that iOS had no manual lock affordance at all (it locked only on scenePhase
  background). Trivial to add because #461 already threads `onLock`; the button
  just calls it. Leading placement so a security control stays visible. Verified
  on-device.
- **Enroll-from-Settings** — needs a master-password re-prompt; deferred on both
  platforms.
- **Managing other devices' slots** — not expressible at any layer (no
  slot-enumeration API in core/bridge/`.udl`).

## Design

### The one real decision: Settings presentation (build-at-tap vs per-render)

macOS builds both VMs **at tap** and shows them in a `.sheet(item:)`. iOS today
builds `makeSettingsViewModel()` **per toolbar render** inside an `if let`
`NavigationLink`
([VaultBrowseScreen.swift:123-133](../../../ios/SecretaryApp/Sources/VaultBrowseScreen.swift#L123-L133)).
Adding the device VM to that path would call `makeDeviceSlotViewModel(port:)` per
render, and its `init` snapshots `port.isEnrolled` — a **Keychain read +
Secure-Enclave probe on the main actor on every SwiftUI re-render**. That is
exactly what the macOS port avoided.

**Chosen approach:** switch the iOS Settings entry from a per-render
`NavigationLink` to a **build-at-tap `.navigationDestination(item:)`**. A `@State`
item holds `{settingsVM, deviceVM}` built once in the tap handler, then pushed.

- Keeps iOS **push** navigation (unlike macOS's sheet) — consistent with the rest
  of the iOS browse stack.
- Pays the `isEnrolled` Keychain read **exactly once**, at tap.
- Mirrors macOS's build-at-tap VM construction semantics.
- Trash stays a `NavigationLink` — only Settings carries the enrollment-probe VM,
  so only Settings needs this. The local inconsistency is deliberate and explained
  in a code comment.

The `.navigationDestination(item:destination:)` overload is iOS 17+; the app
already targets iOS 17+ (it uses the two-parameter `onChange(of:_:)` form at
`VaultBrowseScreen.swift:186`), so the floor is satisfied.

### Component wiring (3 files touched, mirroring macOS)

**1. `SecretaryApp.swift` (`RootView`)** — the `.browse` render arm currently
discards `scoped` (`_`)
([SecretaryApp.swift:218](../../../ios/SecretaryApp/Sources/SecretaryApp.swift#L218)).
Bind it, build the port where the path lives (as `MacRootView` does), and thread
the port + a new `onLock`:

```swift
case .browse(let browseModel, let syncVM, let monitor, let scoped):
    VaultBrowseScreen(
        viewModel: browseModel, syncModel: syncVM,
        deviceSlotPort: CoordinatorDeviceSlotPort(
            coordinator: makePerVaultDeviceUnlock(vaultPath: scoped.pathData).coordinator,
            vaultPath: scoped.pathData),
        onLock: { browseModel.lock(); scoped.end(); route = .select })
        .onDisappear { monitor.stop() }
```

`onLock` reuses the exact lock sequence the `scenePhase == .background` handler
already runs
([SecretaryApp.swift:240-244](../../../ios/SecretaryApp/Sources/SecretaryApp.swift#L240-L244)).
`monitor.stop()` is already covered by the existing `.onDisappear` when the route
changes away, so `onLock` need not repeat it.

**2. `VaultBrowseScreen.swift`** — add `deviceSlotPort: DeviceSlotPort` +
`onLock: () -> Void` to `init`. Replace the Settings `NavigationLink` with a
tap-built `@State` route item + `.navigationDestination(item:)`:

```swift
private struct SettingsRoute: Identifiable {
    let id = UUID()
    let settingsVM: SettingsViewModel
    let deviceVM: DeviceSlotViewModel
}
@State private var settingsRoute: SettingsRoute?

// toolbar Settings button:
Button {
    guard let settingsVM = viewModel.makeSettingsViewModel() else { return }
    settingsRoute = SettingsRoute(
        settingsVM: settingsVM,
        deviceVM: viewModel.makeDeviceSlotViewModel(port: deviceSlotPort))
} label: { Label("Settings", systemImage: "gear") }

// on the NavigationStack:
.navigationDestination(item: $settingsRoute) { route in
    SettingsScreen(viewModel: route.settingsVM,
                   deviceViewModel: route.deviceVM,
                   onForgotten: onLock)
}
```

**3. `SettingsScreen.swift`** — add `deviceViewModel: DeviceSlotViewModel`
(`@StateObject`) + `onForgotten: () -> Void` to `init`. Add a **"This Device"**
`Section` mirroring `MacSettingsView`
([MacSettingsView.swift:118-148](../../../ios/SecretaryMacApp/Sources/MacSettingsView.swift#L118-L148)):
destructive **"Forget This Device"** button → `confirmationDialog`, a
section-scoped `deviceSlotErrorMessage(...)` line, the whole section hidden when
`!deviceViewModel.isEnrolled`. iOS copy uses "Face ID" / "this device". The Save
button gains `|| deviceViewModel.isBusy` (symmetric with macOS: both writers share
the screen, so a settings save must not fire while a revocation is in flight).

The dialog action is the verbatim macOS logic — **do not simplify**:

```swift
Task {
    await deviceViewModel.forget()
    // .forgotten means "this device's credential is gone → lock", NOT
    // "the revocation succeeded". Reached on full success AND on a partial
    // teardown that already killed the enclave key (see DeviceSlotViewModel).
    if deviceViewModel.state == .forgotten { onForgotten() }
}
```

## Security ordering (carried across verbatim — the load-bearing part)

The VM already guarantees re-auth-before-revoke and reaches `.forgotten` on both
full success *and* a partial teardown that killed the enclave key (the `¬G ⟹ ¬D`
discriminator documented in `DeviceSlotViewModel`). The iOS view's only obligation
is to honour `.forgotten → lock`.

Two orderings matter, and both hold by construction:

1. **Re-auth precedes revocation.** Enforced inside `DeviceSlotViewModel.forget()`.
   The gate proves presence via the *same* Secure-Enclave key the revocation
   deletes, and `GraceWindowReauthGate.authorizeWrite` opens with
   `guard authorizer.isEnrolled else { return }` — so gating after the revocation
   would gate nothing.

2. **Revocation completes before `scoped.end()`.** iOS-specific: `forget()` writes
   the `devices/<uuid>.wrap` removal into the **security-scoped** vault folder, so
   the file op must land while the scope is live. It does — `onForgotten()` fires
   only after `await forget()` returns, and `onLock` calls `scoped.end()` after
   `browseModel.lock()`.

**"Dismiss, then lock" on iOS** collapses to just the route change:
`onLock` sets `route = .select` in `RootView`, which unmounts the entire
`.browse` `NavigationStack` — including the pushed `SettingsScreen`. That *is* the
iOS equivalent of macOS's explicit `settingsSheet = nil; onLock()`. No
`@Environment(\.dismiss)` pop is added — an explicit pop racing the route change
would double-animate for no benefit.

## Testing & acceptance

- **No new host tests.** The shared `DeviceSlotViewModel` is already host-tested
  (9 tests) and the issue explicitly says no new VM/port is needed. The existing
  `swift test` suites must stay green.
- iOS app views are **host-untested** (#417) — same posture as macOS D.5.5:
  **compile-proof + manual smoke**. Every code commit is individually
  compile-proven with `bash ios/scripts/build-app.sh`.
- **Gates:**
  - `cd ios/SecretaryVaultAccess && swift test` — fast host suite (323 tests today).
  - `bash ios/scripts/build-app.sh` — iOS app compile-proof.
  - `bash ios/scripts/run-ios-tests.sh` — pure host suites + xcframework +
    SecretaryKit simulator tests.
- Rust workspace / clippy / conformance **not** run — zero `core` / `ffi` change.

### The one pre-merge gate: on-device Face ID smoke

Face ID cannot be exercised by any automated gate. On real hardware, against a
**throwaway vault copy** (never the tracked golden fixture — the app writes
settings into it), with the re-auth **grace window set to 0** first (or wait out
the 2-minute default), verify:

1. Enroll this device (unlock-screen toggle), relaunch, confirm "Unlock with Face
   ID" appears.
2. Unlock → Settings → confirm the **"This Device"** section is present.
3. Forget This Device → confirm dialog → **Face ID prompt** → session locks to the
   unlock screen.
4. The unlock screen now offers the password path with the "Remember this device"
   toggle **restored**, and "Unlock with Face ID" is gone.
5. Re-enroll and confirm Face ID unlock works again (no orphan blocking
   re-enrollment).
6. **Cancel path:** Forget → cancel the Face ID prompt → still enrolled, session
   still unlocked, error shown *in the section*.
7. Open a vault where this device is **not** enrolled → the section is absent.
8. Eyeball the section layout in the grouped `Form` (compile proofs and host
   suites cannot see layout).

## Acceptance criteria (from #461)

- Section present in iOS Settings when enrolled, absent when not.
- Face ID prompt on Forget; session locks back to the unlock screen.
- After forgetting, the unlock screen offers the password path with the "Remember
  this device" toggle restored.
- Cancel path leaves the device still enrolled.
- Existing host suites stay green; no new view model or port.
- On-device proof on real hardware.
