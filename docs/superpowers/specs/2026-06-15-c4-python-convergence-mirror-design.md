# C.4 — Python clean-room convergence mirror (design)

**Date:** 2026-06-15
**Status:** approved (brainstorming) → ready for implementation plan
**Branch:** `feature/c4-python-convergence-mirror`
**Precedes:** writing-plans → subagent-driven TDD execution
**Builds on:** C.4 cross-device convergence conformance (#235, merged `226e6a2`)

## 1. Goal

Extend the clean-room verifier [`core/tests/python/conformance.py`](../../../core/tests/python/conformance.py)
(Python standard library only, **no `secretary-core` import**) to prove that the
C.4 convergence scenarios converge to **identical logical state, independent of
merge order**, under the *pure CRDT merge* — mirroring, inside the clean-room,
the order-independence proof that the Rust harness (`core/tests/convergence.rs`,
#235) established by driving the real sync pipeline.

This extends the repo's core "**docs alone are sufficient**" property from
*single-vault decryption + pairwise merge* to *two-client convergence*: a
clean-room implementation reading only `docs/` reaches the same converged state
the Rust implementation does, and reaches it order-independently.

## 2. Scope

### In scope — CRDT-pure convergence (4 scenarios)

Mirror the scenarios whose convergence is decided entirely by the **pure CRDT
merge**, which is the part specified in the frozen docs (`docs/crypto-design.md`
§11, `docs/vault-format.md`) and already implemented in Python by
`py_merge_block` / `py_merge_record`:

1. **auto-apply** — non-concurrent; one device's clock dominates
   (`IncomingDominates` / `IncomingDominated`). Convergence is trivial but the
   relation-dispatch path is exercised in both orderings.
2. **concurrent disjoint fields** — both devices edit the same record, different
   fields → field-union merge.
3. **LWW field collision** — both edit the *same* field, different `last_mod` →
   later writer wins.
4. **tombstone, AcceptTombstone arm** — one device keeps the record live, the
   other tombstones it with a strictly-later death clock → the record-level
   death clock (`tombstoned_at_ms`) auto-resolves to *tombstoned* in **either**
   order. This is the order-independent, spec-backed arm.

### Out of scope — the veto (deliberate, documented)

The Rust harness's scenario 4 also exercises a **`KeepLocal` veto** arm. The
veto (`prepare_merge`'s veto pass, `commit_with_decisions`, `MergeDecision`)
lives in `core/src/sync/` and is **not specified in any frozen `docs/*.md`** —
it is sync-orchestration user-override logic, not part of the CRDT merge
contract. Mirroring it in the clean-room would require porting undocumented
`core/src` internals into `conformance.py`, weakening the "docs alone
sufficient" property. So:

- The Python mirror covers the **AcceptTombstone / death-clock** resolution
  only (scenario 4a), which is order-independent and spec-backed.
- `KeepLocal` (scenario 4b) is **intentionally not mirrored**. The new
  conformance section documents this boundary in a comment.

> **Deferred follow-up (acknowledged by the user):** eventually the feature set
> (the veto, and other sync-orchestration semantics) should be promoted into the
> spec docs and then into the clean-room. When that happens, the `KeepLocal`
> arm can be added to this mirror. For now this CRDT-pure mirror is the
> agreed baseline. Capture as a backlog item, not done in this slice.

## 3. Fixture — `core/tests/data/convergence_kat.json` (Rust-generated)

A new `#[ignore]`-marked Rust generator test emits the fixture, following the
established `generate_conformance_kat` / `generate_revoke_kat` workflow (CLAUDE.md
"Regenerate … after an intentional protocol change"): generator writes →
human reviews the diff → commit.

### Schema

Reuses the **same plaintext-block schema** that `conflict_kat.json` and the
Python `_normalise_block` already speak (so no new Python parser is needed):

```json
{
  "version": 1,
  "_doc": "C.4 convergence conformance vectors. Each scenario carries two
           concurrent device sides (block plaintext + vector clock) plus the
           golden converged block produced by the real Rust conflict.rs merge.
           Replayed by core/tests/python/conformance.py (clean-room, stdlib
           only) which merges BOTH orderings and asserts (a) order-independence
           and (b) golden-match. Generated by
           `cargo test --release --workspace -- --ignored generate_convergence_kat --nocapture`.",
  "scenarios": [
    {
      "name": "auto_apply",
      "device_a": { "block": { <plaintext block> }, "vector_clock": [ {device_uuid_hex, counter}, ... ] },
      "device_b": { "block": { <plaintext block> }, "vector_clock": [ ... ] },
      "merging_device_a_hex": "<device A uuid hex>",
      "merging_device_b_hex": "<device B uuid hex>",
      "golden": { "block": { <converged plaintext block> } }
    }
    // ... concurrent_disjoint, lww_collision, tombstone_accept
  ]
}
```

Notes:
- `block` uses the `conflict_kat.json` record/field shape: `block_uuid_hex`,
  `block_name`, `block_version`, `schema_version`, `records[]`
  (`record_uuid_hex`, `record_type`, `fields[]` with
  `name`/`value_type`/`last_mod`/`device_uuid_hex`/`value_text`|`value_hex`,
  `tags`, `created_at_ms`, `last_mod_ms`, `tombstone`, `tombstoned_at_ms`,
  `unknown_hex`), and block-level `unknown_hex`.
- **`golden` carries no vector clock.** The merged clock legitimately differs
  by *which device was the merger* (the merge ticks the merger's entry), so it
  is not part of the converged *logical* state — exactly as the Rust harness's
  `LogicalRecord` excludes clocks. Convergence is asserted on the block records,
  not the clock.
- `merging_device_{a,b}_hex` let the Python side tick the correct merger per
  ordering.

### Generator (Rust)

A new test target (e.g. `core/tests/convergence_kat_gen.rs`) that:

1. Builds each scenario's two device-side `BlockPlaintext` + `Vec<VectorClockEntry>`
   **in Rust**, using the same kind of small deterministic builders `conflict.rs`
   uses (`pt` / `record` / `rfield` / `vc`). Inputs are plain constructed data
   (no encryption, no folders) — these scenarios are about the merge layer, so
   plaintext blocks are the right granularity and keep the generator simple.
2. Computes the **golden** by calling the real Rust merge
   (`secretary_core` `merge_block`) on the two sides — the same function
   `conflict.rs` replays — so the golden is the genuine production merge output,
   not hand-typed.
3. Serializes inputs + golden to the schema above. This needs a
   **`Record`/`BlockPlaintext` → JSON serializer** — the *inverse* of
   `conflict.rs`'s `parse_block`/`parse_record`/`parse_field_value`, which exist
   as parsers only. The serializer is the one genuinely new mechanical piece.
4. Writes `core/tests/data/convergence_kat.json`.

The generator is `#[ignore]`-marked (not run in the normal suite); the committed
fixture is what CI consumes.

### Optional Rust always-run guard

If cheap, add a non-ignored Rust test that loads `convergence_kat.json`, replays
both orderings through `merge_block`, and asserts order-independence + golden —
mirroring `conflict.rs`'s `kat_replays_match_rust_merge`. This keeps the fixture
honest on the Rust side too (catches a stale fixture after a merge-semantics
change). **Decision: include it** — it's a few lines reusing the existing
parsers and closes the gap where only Python would notice fixture drift.

## 4. Python replay — new `section_convergence_kat()` in conformance.py

Reuses the **existing** clean-room engine — `py_merge_block`,
`py_clock_relation`, `py_merge_vector_clocks`, `_normalise_block` — with **no new
merge logic**. For each scenario:

1. **Load** `device_a`, `device_b`, `merging_device_{a,b}_hex`, `golden`.
2. **Merge ordering AB** (A canonical, B merges):
   `py_merge_block(local=B.block, local_clock=B.clock, remote=A.block,
   remote_clock=A.clock, merging_device=B_hex)` → `block_ab`.
   `local` = the merger's own side, `remote` = canonical — matching how the
   Rust merger syncs (its own working copy is local, the conflict-copy/canonical
   is remote).
3. **Merge ordering BA** (B canonical, A merges): the mirror, merger = A.
4. **Assert order-independence:**
   `_normalise_block(block_ab["block"]) == _normalise_block(block_ba["block"])`.
   Vector clocks are excluded (they differ by merger, as above).
5. **Assert golden-match:**
   `_normalise_block(block_ab["block"]) == _normalise_block(golden["block"])` —
   Python's independent merge reaches the same state Rust's real merge produced.
6. Emit `PASS  convergence_kat.json {name!r}` lines, matching the existing
   section style; wire the section into `main()` and the overall pass/fail
   gate.

## 5. Why this is sound

- **Order-independence over merged records is the CRDT commutativity property** —
  already proptest-proven in Rust (`conflict.rs` commutativity/associativity
  properties). `convergence_kat.json` pins *named scenario instances* of that
  property plus the converged golden, in the clean-room.
- **Clocks correctly excluded.** The only legitimate cross-ordering difference
  is the merger's clock tick; excluding the clock from the convergence
  comparison is exactly what the Rust `LogicalRecord` projection does.
- **Clean-room preserved.** The generator (Rust) produces the fixture; the
  verifier (Python) uses only the standard library and the spec. No
  `secretary-core` import is added to `conformance.py`.

## 6. Testing & TDD

Per the project's TDD discipline:

- **Python side, test-first:** write `section_convergence_kat()` to assert
  against `convergence_kat.json` *before* the fixture exists → run
  `conformance.py` → red (missing fixture). Then add the generator, generate the
  fixture → green. (Mirrors how a KAT consumer is written ahead of its vector.)
- **Rust generator + always-run guard:** standard `cargo test` TDD — guard test
  red against absent/mismatched fixture, then generator makes it green.
- Scenarios are small and deterministic; no randomness needed (these are merge
  vectors, not crypto material — the "random crypto values" rule does not apply
  to merge-layer plaintext fixtures, mirroring `conflict_kat.json`).

## 9. Risks / open notes

- **New `Record`→JSON serializer must round-trip with `conflict.rs`'s parsers.**
  Verify by feeding a generated block back through `parse_block` (or just trust
  the always-run guard, which re-parses the committed fixture). Keep the two in
  lockstep — a field added to the schema needs both sides updated.
- **Whole-record-overwrite edit semantics** carry over from the Rust harness:
  a single-field edit drops sibling fields on that device's own side before the
  merge (so e.g. a seed field is not retained). Author the scenario inputs to
  reflect this honestly (don't expect a dropped seed field in the golden).
- **`unknown_hex` hex-case:** the existing `section5` self-test already guards
  `py_merge_unknown_map` case-insensitivity; if a scenario carries unknown keys,
  keep hex lowercase to match the generator's output.
- **Veto deferred** (see §2) — the one acknowledged feature gap; backlog, not
  this slice.
