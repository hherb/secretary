# D.5.3 — Native macOS record + block mutation UI (design)

**Date:** 2026-07-18
**Sub-project:** D.5 (native macOS SwiftUI client, [ADR 0011](../../adr/0011-macos-native-swiftui.md))
**Depends on:** D.5.2 (native read-only viewer, shipped 2026-07-17)
**Status:** design approved; implementation plan to follow

## 1. Summary

Grow the macOS app from the D.5.2 **read-only** viewer into a **read-write** client:
add / edit / delete / restore / move **records** and create / rename **blocks**, plus
a Show-deleted toggle — reaching parity with the iOS *inline browse* mutation surface.

This is a **presentation port**, exactly like D.5.2. The entire mutation surface
already exists — and is host-tested — in the shared, macOS-compatible
`SecretaryVaultAccessUI` view models (`VaultBrowseViewModel`, `RecordEditViewModel`).
D.5.3 writes macOS-idiomatic SwiftUI views over them. There is **no** change to
`core`, FFI / `.udl`, `FfiVaultError`, the on-disk format, or the shared view models.
No new FFI surface, no new dependency.

`MacRootView.enterBrowse` already injects `trashPort` + `settingsPort` into the browse
VM, so no wiring change is needed there either.

The Tauri macOS build remains the shipping macOS client; D.5.3 **coexists** and does
not cut over (per ADR 0011).

## 2. Scope

### In scope (all over `VaultBrowseViewModel` / `RecordEditViewModel`)
- **Record add** — `RecordEditViewModel(mode: .add)` → `appendRecord`.
- **Record edit** — `RecordEditViewModel(mode: .edit)` (pre-loaded via `load(record:)`)
  → `editRecord`.
- **Record delete (tombstone)** — `delete(record:)` behind an explicit confirm.
- **Record restore (resurrect)** — `restore(record:)` on a tombstoned record (only
  reachable while Show-deleted is on).
- **Record move** — `startMoveRecord` / `confirmMove` via a target-block picker,
  gated on `hasMoveTargets` (hidden when the vault has < 2 live blocks).
- **Block create** — `startCreateBlock` / `confirmBlockName`.
- **Block rename** — `startRenameBlock` / `confirmBlockName`, with the **warn-but-allow**
  duplicate-name affordance (`blockNameCollides` → "Save" relabels to "Save anyway").
- **Show-deleted toggle** — binds `showDeleted`; the Rust gate withholds tombstoned
  records, so toggling **re-reads** the block (the client never filters withheld data).

### Non-goals (explicitly deferred to D.5.4+)
- **Trash-block view** (list trashed blocks → restore / purge) over `TrashViewModel`.
- **Settings screen** (retention window, reveal grace, "Remember this Mac" toggle)
  over `SettingsViewModel`.
- Sync UI + folder-change monitor (unchanged from D.5.2 — re-open to pick up external
  changes).
- Share / contacts / revoke.
- Create-vault wizard.
- App Sandbox + security-scoped bookmarks + notarization + Mac App Store.
- Intel (`x86_64`) — Apple Silicon only, matching D.5.1/D.5.2.

## 3. Interaction idiom — native macOS (approach A)

iOS drives row mutation with **swipe actions**; that is an iOS convention. On macOS
this slice uses the platform-native equivalents, consistent with D.5.2's already-native
direction (three-column `NavigationSplitView` + window toolbar):

- **Context menus (right-click)** on rows carry the per-item actions:
  - Record row (live): **Edit**, **Move…** (only when `hasMoveTargets`), **Delete**.
  - Record row (tombstoned, Show-deleted on): **Restore**.
  - Block sidebar row: **Rename…**.
- **Window toolbar** carries the create actions:
  - **Add record** — shown only when a block is selected.
  - **New block** — always available.
  - (**Lock** already exists from D.5.2.)
- **Show-deleted** is a `Toggle` in the records column (header/toolbar of the middle
  column).
- **Delete** is confirmed via `.confirmationDialog` (default is non-destructive
  Cancel; no silent data loss).
- Every write disables its trigger while `isWriting` is true (parity with iOS).

Rejected alternatives: mirroring iOS `.swipeActions` (undiscoverable with a mouse,
reads as an iOS port) and inline per-row buttons (clutters a dense 3-column list).

## 4. Architecture / components

All work is in `ios/SecretaryMacApp/Sources/`. macOS deployment target is **13.0**,
Swift 6 language mode — the same API floors D.5.2 already documents in `MacBrowseView`
(single-parameter `onChange(of:)`, trailing-closure `Button { } label:` instead of the
macOS-14 `Button(_:systemImage:)`).

### New files
- **`MacRecordEditView.swift`** — the add/edit **sheet** over `RecordEditViewModel`.
  A sheet (not inline detail-column editing) matches iOS's explicit Save/Cancel →
  re-read-on-done flow and keeps edit decoupled from the detail column's reveal/mask
  model. macOS-idiom `Form`:
  - `TextField` for record type; tag rows; field rows with a segmented `Picker`
    (Text / Bytes-hex) and `SecureField` for `.text` values, plain `TextField` for hex.
  - **macOS differences from iOS:** no `.textInputAutocapitalization` (iOS-only); no
    `.onDelete` swipe-to-remove (there is no List edit mode) — field/tag removal uses an
    explicit per-row delete button. `.autocorrectionDisabled()` is available on macOS 12+.
  - Save is disabled on `loadFailed || committed || isWriting`; `onChange(committed)`
    dismisses + triggers the browse re-read (mirrors iOS `RecordEditScreen`).
- **`MacBlockCrudViews.swift`** — two sheets mirroring iOS `BlockCrudViews.swift`:
  - `MacBlockNameSheet` — create/rename, live `blockNameCollides` warning, confirm
    button relabel "Save" → "Save anyway"; disabled while `isWriting`; error shown inline.
  - `MacMoveTargetPicker` — lists every block except the source; a row → `confirmMove`.

### Extended file
- **`MacBrowseView.swift`** (208 lines today) gains: context menus on record + block
  rows, toolbar **Add record** / **New block**, the Show-deleted toggle, a delete
  `.confirmationDialog`, and the sheet/dialog presentation state (`editSession`,
  `blockNameField`, `movingItem`, `recordPendingDelete`). If it crosses ~400 lines the
  record-row builder moves to a `MacBrowseView+Rows.swift` extension (per the 500-line
  rule). The existing read-only machinery (reveal/mask/copy/auto-clear, resign-active
  hide, window-close wipe) is **untouched**.

### Data flow (unchanged invariants)
Every mutation routes through the VM's existing `reauthedWrite` / `RecordEditViewModel.commit`,
which awaits the **write re-auth gate** (Touch ID or password) **before** the write and
re-reads (`refresh()` / `loadBlocks()`) after. The macOS views never bypass the gate.
Same verify-before-decrypt / re-auth-before-write guarantees as iOS.

## 5. Testing / TDD

The mutation **logic** is already host-tested in `swift test` and is reused verbatim
by these views — no new algorithmic surface is introduced:
- `RecordEditViewModelTests` — add / edit / validation / load-failure / gate refusal.
- `VaultBrowseViewModelBlockCrudTests`, `…MoveTests`, `…BlockNameWarnTests`,
  `…DeletedTests` — create / rename / tombstone / resurrect / move / collision /
  show-deleted re-read.

This slice deliberately adds **no new pure logic**. If a non-trivial pure decision
emerges during implementation (e.g. a formatting/title helper), it is placed in
`SecretaryVaultAccess` with its own unit test rather than inlined in a view (per the
pure-functions-in-reusable-modules rule).

App-level correctness is the **`ios/scripts/run-macos-tests.sh` compile proof**
(XcodeGen + `xcodebuild` → `SecretaryMac.app` BUILD SUCCEEDED with the new mutation UI
linked in), consistent with D.5.1/D.5.2 and the current "no ViewInspector/UI-test
target yet" posture (that infra decision is tracked separately, cf. #417).

## 6. Security / error handling

- No new FFI, secret-handling path, or on-disk write path — all writes go through the
  frozen-`core` bridge orchestrators the shared VMs already call.
- **Reveal-during-edit:** `RecordEditViewModel.load` reveals existing field plaintext
  into editable rows; this reuses the VM's existing behavior with the same best-effort
  Swift-COW zeroize caveat as iOS ([[project_secretary_swift_zeroize_cow_noop]]) — no
  new concern, no new secret lifetime.
- **Copy / reveal / auto-clear** already live in `MacBrowseView` (concealed-type
  pasteboard hint + `changeCount`-guarded clear) and are not touched.
- Typed `VaultAccessError` surfaces inline in each sheet / the browse overlay via
  `error.localizedDescription` (the D.5.2/#454 `LocalizedError` conformance already
  produces user-safe, anti-oracle copy).

## 7. Acceptance criteria

1. `cd ios/SecretaryVaultAccess && swift test` — green (unchanged suite).
2. `cd ios/SecretaryDeviceUnlock && swift test` — green (unchanged suite).
3. `bash ios/scripts/run-macos-tests.sh` — `SecretaryMac.app` **BUILD SUCCEEDED** with
   the new mutation UI compiled and linked; D.5.1 device-unlock acceptance still PASS.
4. Manual (non-CI) macOS smoke, exercised at least once by the author: add a record,
   edit it, delete + restore it (Show-deleted on), move it between blocks, create a
   block, rename a block, observe the duplicate-name warning. (Documented, not a gate —
   there is no macOS UI-test target.)
5. No diff under `core/`, `ffi/`, `*.udl`, or the on-disk format; no new dependency;
   no change to the shared `*ViewModel.swift` files.
