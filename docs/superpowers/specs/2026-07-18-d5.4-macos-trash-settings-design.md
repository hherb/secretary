# D.5.4 — native macOS Trash view + Settings screen (design)

**Date:** 2026-07-18
**Sub-project:** D.5 (native macOS SwiftUI client, ADR 0011)
**Branch:** `feature/d5-macos-trash-settings` · worktree `.worktrees/d5-macos-trash-settings`
**Predecessor:** D.5.3 (record + block mutation UI, PR #457, merged)

## Summary

Grow the native macOS client from D.5.3 (record + block mutation) by adding the two
remaining browse-adjacent surfaces the iOS app already ships: a **Trash view** and a
**Settings screen**. Both are driven entirely through the **already-host-tested shared
view models** — `TrashViewModel` and `SettingsViewModel` — built via the existing
`VaultBrowseViewModel.makeTrashViewModel()` / `makeSettingsViewModel()` factories, which
already return non-nil on macOS because `MacRootView.enterBrowse` injects `trashPort` +
`settingsPort` into the browse VM.

This is a **pure view-layer slice**: no `core` / `.udl` / `FfiVaultError` / on-disk-format
/ view-model change — the same shape as D.5.3. The macOS work is SwiftUI over VMs that are
already covered by host tests.

## Goals

- macOS **Trash sheet**: list trashed blocks; per-block **Restore** and **Delete forever**;
  **Empty trash**; **Run retention now** — every destructive op behind the shared re-auth
  write gate (the VM already enforces this).
- macOS **Settings sheet**: edit the per-vault **retention window** (days) and **re-auth
  grace** (minutes); Save routes through the shared gate and retargets the live gate on
  success — all inside `SettingsViewModel`; the view never bypasses it.
- Present both via **toolbar buttons → modal sheets** on `MacBrowseView`, reusing the
  D.5.3 `Identifiable`-wrapper `.sheet(item:)` pattern already in that file.
- Keep every file within the project's size discipline (extract when crossing the
  established 400-line threshold for `MacBrowseView`).

## Non-goals (explicit)

- **"Remember this Mac" device-slot toggle in Settings → deferred to D.5.5.** It is *not*
  covered by the shared `SettingsViewModel` (which is retention + grace only); enrolling
  from an already-open session needs a password re-prompt, and both enroll and un-enroll
  touch the Secure-Enclave / Keychain adapter — i.e. it is **not host-testable**, unlike
  the rest of D.5.4. Device enrollment continues to happen at unlock time via
  `MacUnlockView`'s "Remember this Mac with Touch ID" toggle, which already works.
- **No swipe idiom.** macOS uses context menus + toolbar (the D.5.3 decision); swipe-to-act
  is undiscoverable with a mouse.
- **No `core` / FFI / on-disk-format / view-model change.** If any of these appear needed,
  stop — the slice has drifted out of scope.

## Architecture

### New files

- `ios/SecretaryMacApp/Sources/MacTrashView.swift` — the Trash sheet. Holds a
  `@StateObject TrashViewModel` (initialised from the factory-built VM, mirroring iOS
  `TrashScreen`).
- `ios/SecretaryMacApp/Sources/MacSettingsView.swift` — the Settings sheet. Holds a
  `@StateObject SettingsViewModel` (mirroring iOS `SettingsScreen`).

### Modified file

- `ios/SecretaryMacApp/Sources/MacBrowseView.swift`
  - Two new primary-action toolbar buttons — **Trash** (`trash` icon) and **Settings**
    (`gear` icon) — each rendered only when its factory returns non-nil (mirrors iOS
    `VaultBrowseScreen`). Toolbar order: create actions (Show-deleted, Add record, New
    block) → Trash → Settings → Lock. Six primary items total (approved).
  - Two `.sheet(item:)` presentations, each keyed on a small `Identifiable` wrapper
    (`TrashSheetItem` / `SettingsSheetItem`) holding the freshly-built child VM — the same
    lifetime pattern as the existing `EditSession`. The VM is built at button-tap (matching
    iOS, which builds it in the `NavigationLink` label site), so each open reflects current
    vault state via the child VM's own `.onAppear` load.

### Targeted refactor (size discipline)

`MacBrowseView.swift` is at 377 lines against an established 400-line extract threshold
(noted in the D.5.3 handoff). Adding the Trash/Settings wiring crosses it, so extract the
**field-detail column** — `fieldList`, `fieldRow`, `display`, `copyToPasteboard`, and the
`WindowAccessor` helper (≈90 lines) — into a new **`MacBrowseView+Fields.swift`**. This is
one cohesive concept (the third split-view pane + its pasteboard/reveal plumbing), drops
the core file to ≈290 lines, and leaves clear headroom for the new wiring. It is a pure
mechanical extraction (no behaviour change), compile-proven by the build.

## View designs

### MacTrashView (mirrors `TrashScreen`, macOS idiom)

- **Purge-notice banner** at the top when `viewModel.purgeNotice != nil` (warning →
  orange, else secondary) — reuses the shared `PurgeNotice` surface.
- **`List`** of `viewModel.entries` (`[TrashedBlockInfo]`); each row shows the block name
  (or "block" when empty) and "trashed \<when\>" via the shared `formatTrashedWhen`. Empty
  state: "Trash is empty."
- **Per-row context menu** (not swipe): **Restore** (`viewModel.restore(uuid:)`) and
  **Delete forever** (opens a `.confirmationDialog`, then `viewModel.purge(uuid:)`), both
  `.disabled(viewModel.isWriting)`.
- **Sheet toolbar / controls**: **Run retention now** (nested retention sheet reusing
  `previewRetention` / `runRetention` / `clearPreview` + the shared `retentionSummary`),
  **Empty trash** (`.confirmationDialog` with the shared `emptyTrashConfirmBody`), and a
  **Done** button to dismiss the sheet.
- `.onAppear { viewModel.load() }`.

The nested retention sheet mirrors iOS `RetentionSheet`: `previewRetention()` on appear,
a "Purge N items" destructive button when the preview is non-empty, `clearPreview()` on
disappear.

### MacSettingsView (mirrors `SettingsScreen`, macOS idiom)

- **`Form`** with `.formStyle(.grouped)` (macOS 13+).
- Two `LabeledContent` rows — **Delete trash after** (days) and **Re-auth grace**
  (minutes) — each a `TextField(value:format:.number)` bound via
  `setRetentionDays(_:)` / `setGraceMinutes(_:)`. **macOS diff:** drop
  `.keyboardType(.numberPad)` (iOS-only; same class of platform diff as D.5.3 dropping
  `.textInputAutocapitalization`).
- **Banner** (`viewModel.banner`) and **error** (`settingsErrorMessage(viewModel.error)`)
  rows; section footers show the valid ranges (`retentionDaysRange` / `graceMinutesRange`).
- **Save** button → `viewModel.save()`, `.disabled(viewModel.isWriting)`; **Done** to
  dismiss. `.onAppear { viewModel.load() }`.
- The load-bearing security ordering (gate against the *current* grace window, retarget the
  live gate only on a successful save that actually changed the grace) is entirely inside
  `SettingsViewModel` — the view only binds controls and calls `save()`.

## API-floor constraints (deploymentTarget macOS 13.0)

- Use the trailing-closure `Label` form (not the macOS-14 `Button(_:systemImage:action:)`).
- Use single-param `onChange(of:_:)` (not the macOS-14 two-param overload).
- `LabeledContent`, `.formStyle(.grouped)`, `.confirmationDialog` are all macOS 13-safe.

(All three constraints are already documented in `MacBrowseView.swift` from D.5.3.)

## Testing & acceptance

Pure view layer over VMs that already have host coverage — `TrashViewModelTests` and
`VaultBrowseViewModelSettingsFactoryTests` — so **no new host tests are required**. The
`MacBrowseView+Fields` extraction is a behaviour-preserving refactor, compile-proven by
the build.

Acceptance mirrors D.5.3:

- `bash ios/scripts/run-macos-tests.sh` → `SecretaryDeviceUnlock` /
  `SecretaryVaultAccess` / `SecretaryKit` host suites green (0 failures) and
  `SecretaryMac.app **BUILD SUCCEEDED**`.
- **Manual GUI smoke** (author-run, same manual-only posture as D.5.3 / #161): open Trash →
  Restore a block / Delete-forever (confirm) / Empty trash (confirm) / Run retention now;
  open Settings → edit retention + grace → Save (the write-reauth gate fires) → reopen
  Settings and confirm the persisted values round-trip. Every destructive Trash op and the
  Settings Save must prompt the write-reauth gate.

## Risks / open items

- **Toolbar crowding** — six primary toolbar items. Approved; matches the iOS set.
- **Nested sheet** (retention within the Trash sheet) works on macOS; verify dismissal
  ordering during the smoke (closing the retention sheet must not dismiss the Trash sheet).
- **Scope discipline** — the device-slot toggle stays out (D.5.5). If implementation starts
  reaching for the enclave/Keychain or a password re-prompt, the slice has drifted.
