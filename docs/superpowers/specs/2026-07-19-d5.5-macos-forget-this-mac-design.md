# D.5.5 — native macOS "Forget this Mac" device-slot revocation (design)

**Date:** 2026-07-19
**Slice:** Sub-project D, D.5.5
**Branch:** `feature/d5.5-macos-forget-this-mac`
**Predecessor:** D.5.4 (PR #458, `main` @ `3c8f083`)

## Summary

The macOS Settings screen gains a **"This Mac"** section with a single destructive
action — **Forget This Mac** — that revokes this device's per-device wrap slot
(ADR 0009), clears the Secure-Enclave key and enrollment metadata, and locks the
session.

Today device enrollment is strictly **one-way** on both platforms: the
`Toggle("Remember this Mac with Touch ID")` at
[`MacUnlockView.swift:60`](../../../ios/SecretaryMacApp/Sources/MacUnlockView.swift)
is rendered only when `!biometricEnrolled`, so once enrolled it disappears and can
never be switched off. The capability to undo it exists and is host-tested all the
way down — `DeviceUnlockCoordinator.disenroll(vaultPath:)`
([`DeviceUnlockCoordinator.swift:65`](../../../ios/SecretaryDeviceUnlock/Sources/SecretaryDeviceUnlock/DeviceUnlockCoordinator.swift))
— but **no shipping UI on either platform reaches it**. This slice is therefore UI
plus orchestration; it introduces no new cryptography and no new FFI surface.

The one security-relevant property in the slice is an **ordering constraint**
(§"The ordering constraint" below), and that is what the host tests pin.

## Goals

1. A user who enrolled this Mac for Touch ID unlock can revoke it from Settings.
2. Revocation is re-auth gated, confirmed, and **all-or-nothing from the user's
   point of view** — a failure at any step leaves the device still enrolled.
3. The session **locks** on success, so the write-reauth gate can never be
   silently downgraded mid-session (§"The ordering constraint").
4. The orchestration is **host-tested** — no security-relevant ordering lives in
   un-covered app-target view code.
5. The shared port + view model are platform-neutral, so iOS parity is a small
   follow-up rather than a rewrite.

## Non-goals (explicit)

- **Enroll-from-Settings.** `add_device_slot` runs a second Argon2id and needs the
  master password, which an open session does not hold. That needs a
  password re-prompt sheet and is not host-testable end-to-end. Enrollment stays
  on the unlock screen. (Deferred, as it was from D.5.4.)
- **Managing *other* devices' slots.** There is **no slot-enumeration API at any
  layer** — not in `core/src/vault/device_slot.rs` (which exposes exactly
  `add_device_slot` / `open_identity_with_device_secret` / `remove_device_slot`),
  not in the bridge, not in the `.udl`, not on `VaultDeviceSlotPort`. A "manage my
  enrolled devices" list is not expressible without new core + bridge + udl
  surface. Only **self**-removal is in scope.
- **iOS wiring.** The shared VM lands in this slice; adding the section to the iOS
  `SettingsScreen` is a follow-up issue.
- **Render-layer tests for the macOS views.** The standing structural D.5.x gap
  (see §Risks). Unchanged by this slice.
- **Any change to `core`, the `.udl`, `FfiVaultError`, or the on-disk format.**

## The ordering constraint

This is the load-bearing detail of the slice and the reason it is not a one-liner.

The write-reauth gate proves presence by driving **the same per-vault Secure-Enclave
key** that device unlock uses. `makeRetargetableReauthGate` builds an
`EnclaveBiometricAuthorizer` over `makePerVaultDeviceUnlock(vaultPath:).enclave`
([`RetargetableGateFactory.swift:30-31`](../../../ios/SecretaryKit/Sources/SecretaryKit/VaultAccess/RetargetableGateFactory.swift)),
and `GraceWindowReauthGate.authorizeWrite` opens with:

```swift
guard authorizer.isEnrolled else { return }            // no SE key: no gate
```

([`GraceWindowReauthGate.swift:33`](../../../ios/SecretaryVaultAccess/Sources/SecretaryVaultAccessUI/GraceWindowReauthGate.swift))

`disenroll` calls `enclave.clear()`, which deletes that SE key. Therefore:

- **Re-auth MUST be requested BEFORE the forget.** Afterwards `isEnrolled` is
  false, the guard returns early, and the gate authorizes nothing. Gating after
  the forget would be gating a no-op.
- **The session MUST lock after a successful forget.** Otherwise the remainder of
  the session runs with `authorizeWrite` as a permanent no-op — every subsequent
  record edit, trash purge, and settings save silently ungated, with no
  user-visible signal. That is a security downgrade the user did not ask for and
  cannot observe.

Locking is also the honest UX: the user just revoked this Mac's credential, so
returning to the unlock screen is the expected outcome. The next unlock is
password-only, and the gate is then correctly absent because the vault genuinely
is not enrolled — the same posture as a vault that was never enrolled.

## Architecture

Follows the established `SettingsPort` / `TrashPort` idiom: a narrow port protocol
in the pure package, a host-tested view model in the UI package, a fake in the
testing package, and a thin adapter in `SecretaryKit`.

`SecretaryVaultAccess` and `SecretaryDeviceUnlock` are **deliberately disjoint**
packages (neither declares a dependency on the other); they are joined only at the
app target. The port is what lets the shared, host-tested VM drive a device-slot
operation without creating that package edge.

| Path | Contents |
|---|---|
| `ios/SecretaryVaultAccess/Sources/SecretaryVaultAccess/DeviceSlotPort.swift` | `public protocol DeviceSlotPort: Sendable` — `var isEnrolled: Bool { get }`, `func forgetThisDevice() throws` |

| `ios/SecretaryVaultAccess/Tests/SecretaryVaultAccessUITests/DeviceSlotViewModelTests.swift` | Host tests (§Testing) |

The port names the operation `forgetThisDevice()`, **not** `removeDeviceSlot(uuid:)`.
Since no enumeration API exists, self-removal is the only expressible operation and
the name must not imply otherwise.

`CoordinatorDeviceSlotPort` captures `vaultPath` at construction so the port stays a
2-member protocol with no argument threading, and so the pure package never learns
what a vault path is.

### Port plumbing

`MacBrowseView` today holds only `viewModel` and `onLock` — it has **no vault
path**, so it cannot build a coordinator. `MacRootView` does: `enterBrowse` already
receives `scoped`, and `enterUnlock` already calls `makePerVaultDeviceUnlock`.

The port is therefore constructed in `MacRootView` and injected:

```swift
MacBrowseView(viewModel: browseVM,
              deviceSlotPort: CoordinatorDeviceSlotPort(
                  coordinator: makePerVaultDeviceUnlock(vaultPath: scoped.pathData).coordinator,
                  vaultPath: scoped.pathData),
              onLock: { … })
```

This keeps `MacBrowseView` ignorant of coordinators and enclaves — it just forwards
a `DeviceSlotPort` — and keeps composition in the root view, where it already lives.

**Scope lifetime:** the port captures `vaultPath` from a `ScopedVaultPath` whose
`end()` is called in `onLock`. The forget completes strictly before the lock is
signalled, so the port is only ever exercised while the security-scoped access is
live. The plan must not reorder these.

### Composition

```swift
// VaultBrowseViewModel — mirrors makeTrashViewModel() / makeSettingsViewModel()
public func makeDeviceSlotViewModel(port: DeviceSlotPort) -> DeviceSlotViewModel {
    DeviceSlotViewModel(port: port, gate: gate)
}
```

The caller supplies the **port** (only the app target can build a
`DeviceUnlockCoordinator`); the factory supplies the **gate** from its own private
`gate` property. This is the same reason `makeSettingsViewModel()` derives its gate
by downcast rather than by injection: it guarantees the forget path shares the one
gate instance every other writer uses, and they cannot drift apart.

Unlike the other two factories this one is **non-optional** — it needs no port
downcast and no `settingsPort`. Whether the section renders is decided by
`isEnrolled`, not by the factory returning `nil`.

### View-model surface

```swift
@MainActor
public final class DeviceSlotViewModel: ObservableObject {
    public enum State: Equatable { case idle, forgotten }

    @Published public private(set) var state: State = .idle
    @Published public private(set) var isBusy = false
    @Published public private(set) var error: VaultAccessError?

    /// Snapshotted at init — a Keychain read plus an SE key probe, not per-render.
    public let isEnrolled: Bool

    public init(port: DeviceSlotPort, gate: WriteReauthGate)

    /// Non-throwing: failures fold into `error`, matching `SettingsViewModel.save()`.
    public func forget() async
}
```

The gate is a plain `WriteReauthGate`, **not** a `RetargetableReauthGate` — forgetting
never changes the grace window, so it has no reason to retarget. That is why this
factory needs no downcast and can be non-optional, unlike `makeSettingsViewModel()`.

## Data flow

```
Settings sheet
  └─ "This Mac" section            (rendered only when port.isEnrolled)
       └─ "Forget This Mac"        (.destructive)
            └─ confirmationDialog  ("Forget" / "Cancel")
                 └─ DeviceSlotViewModel.forget()
                      1. guard !isBusy                       // double-tap guard
                      2. try await gate.authorizeWrite(...)  // Touch ID — gate LIVE here
                      3. try port.forgetThisDevice()         // slot + enclave + metadata
                      4. state = .forgotten
                           └─ view observes .forgotten
                                └─ dismiss sheet, then onForgotten()
                                     └─ MacBrowseView → onLock()
```

`forgetThisDevice()` is synchronous `throws`, matching
`DeviceUnlockCoordinator.disenroll`. **No off-main offload is needed**: it performs
one file delete (`remove_device_slot` is a plain `std::fs::remove_file` on
`devices/<uuid>.wrap`, needing neither the master password nor an open vault) plus
two Keychain deletes. It runs no Argon2id, which is the criterion that forces
offload elsewhere in this codebase.

**Sheet-then-lock ordering.** The view sets its sheet-item `@State` to `nil`
*before* signalling the lock. `VaultBrowseViewModel.lock()` nils
`blockNameDialog`/`movingRecord` but cannot see view-local `@State` sheets — the
latent interaction documented in D.5.4 — so the view must tear its own sheet down.
Dismissing first also keeps the `willClose` window-identity filter (load-bearing for
sheets, since each sheet is its own `NSWindow`) from seeing a close it would
misattribute.

## View design

A third `Section("This Mac")` in the existing grouped `Form`, after "Trash" and
"Security":

```
┌─ This Mac ────────────────────────────────────┐
│  [ Forget This Mac ]                (.destructive)
│                                                │
│  Removes Touch ID unlock for this vault on     │
│  this Mac. You'll need your master password    │
│  to unlock, and can turn Touch ID back on      │
│  then. Other devices are unaffected.           │
└────────────────────────────────────────────────┘
```

The footer states all three things a user needs to predict the outcome: what is
lost, how to recover it, and that the blast radius is this Mac only.

**Hidden entirely when not enrolled.** There is nothing to forget, and a
permanently-disabled control is noise. `isEnrolled` is **snapshotted at VM init**,
not read per-render — it costs a Keychain read plus an SE key probe, and
`MacSettingsView`'s VM is already built at toolbar tap rather than per-render for
exactly this reason (`SettingsViewModel` snapshots its bounds the same way).

**Confirmation** uses `confirmationDialog`, matching the destructive-op idiom
`MacTrashView` established in D.5.4 (delete-forever, empty-trash).

**In-flight and error states** reuse the existing message area at the top of the
Form: the button is `.disabled(viewModel.isBusy)` alongside the existing
`viewModel.isWriting` disable on Save, and a failure renders through the same
`settingsErrorMessage(...)` treatment. No new message-area concept.

## Error handling

Failures fold into `VaultAccessError` per the `SettingsViewModel` idiom:

| Failure | Mapping | Effect |
|---|---|---|
| Gate refusal / biometric cancel | `.reauthFailed` | Nothing removed, still enrolled, no lock |
| `forgetThisDevice()` throws | `.other` | Nothing forgotten from the VM's view, no lock |
| Non-`VaultAccessError` throw | `.reauthFailed` / `.other` as above | Same |

Any failure leaves `state != .forgotten`, so the session is **not** locked and the
user can retry or cancel. Nothing is left half-revoked from the user's perspective.

`disenroll` is itself resilient to partial prior failure: it tolerates
`VaultSlotError.deviceSlotNotFound` as already-gone and then clears the enclave and
metadata **unconditionally**, so a retry after a mid-flow failure converges rather
than stranding an orphan. This is pre-existing, host-tested behaviour
(`DeviceUnlockCoordinatorTests` lines 186–240) that this slice relies on and does
not modify.

## Testing & acceptance

### Host tests (`swift test`, no simulator, no FFI)

New `DeviceSlotViewModelTests.swift`, using `FakeDeviceSlotPort` +
`FakeWriteReauthGate` and the ordering-recorder pattern already used by
`SettingsViewModelTests`:

1. **`testForgetAuthorizesBeforeForgetting`** — the ordering constraint. A shared
   recorder asserts `authorizeWrite` is observed **strictly before**
   `forgetThisDevice`. This is the test that protects §"The ordering constraint";
   it must fail if the two calls are ever transposed.
2. **`testGateRefusalDoesNotForget`** — gate throws ⇒ `forgetCallCount == 0`,
   `state != .forgotten`, error set.
3. **`testPortFailureDoesNotReachForgotten`** — port throws ⇒ `state != .forgotten`
   (so the caller never locks), error set.
4. **`testSuccessReachesForgotten`** — happy path ⇒ `state == .forgotten`,
   `forgetCallCount == 1`, no error.
5. **`testConcurrentForgetIsGuarded`** — two overlapping `forget()` calls ⇒
   `forgetCallCount == 1`.
6. **`testNotEnrolledIsReflectedInViewModel`** — `isEnrolled == false` surfaces so
   the view can hide the section.
7. **`testFactoryPassesBrowseGate`** — `makeDeviceSlotViewModel(port:)` wires the
   browse VM's own gate (asserted by observing the gate double).

### Existing suites

`SecretaryVaultAccessUITests`, `SecretaryDeviceUnlockTests`, and
`SecretaryKitTests` must stay green. No existing view model changes behaviour.

### Manual on-hardware proof (required before merge)

Automated gates cannot exercise Touch ID or the Secure Enclave, and cannot see
layout. Required, from a signed Xcode run against a **throwaway vault copy**
(never the tracked golden fixture):

1. Enroll this Mac (unlock screen toggle), then relaunch and confirm "Unlock with
   Touch ID" appears.
2. Unlock, open Settings → confirm the **"This Mac"** section is present.
3. Forget This Mac → confirm dialog → **Touch ID prompt appears** → session locks
   back to the unlock screen.
4. Confirm the unlock screen now offers the **password** path with the "Remember
   this Mac" toggle **restored** (i.e. `biometricEnrolled` is false again), and the
   "Unlock with Touch ID" button is gone.
5. Re-enroll and confirm Touch ID unlock works again — proves the revocation left
   no orphan blocking re-enrollment.
6. **Cancel path:** Forget This Mac → cancel the Touch ID prompt → confirm the
   device is **still enrolled** (section still present, session still unlocked).
7. Open a vault where this Mac is **not** enrolled → confirm the section is absent.
8. Eyeball the section's layout in the grouped `Form` — D.5.4 shipped a
   double-labelling bug that only a screenshot caught.

## Risks / open items

- **No render-layer coverage for the macOS view.** Structural and pre-existing
  across D.5.x (tracked for mobile as #417). `run-macos-tests.sh` compile-proves
  the app but runs no test action against it, so the new section is smoke-verified
  once and then regression-unprotected. The mitigation is that all the
  security-relevant logic is in the host-tested VM, not the view.
- **The lock signal crosses an untested seam.** `state == .forgotten → dismiss →
  onLock()` lives in app-target view code. The VM half is tested (test 3 pins that
  a failure never reaches `.forgotten`), but that the view actually locks on
  `.forgotten` is manual-proof only (step 3 above).
- **Revocation is local-only by construction.** `remove_device_slot` deletes this
  Mac's `devices/<uuid>.wrap`. If the vault folder is a synced cloud folder, the
  deletion propagates on the next sync like any other change — it is not an
  instantaneous global revocation, and a device offline with a cached secret is
  outside this slice's threat model (unchanged from ADR 0009).
- **`isEnrolled` is a snapshot.** If enrollment changed in another process between
  opening Settings and tapping Forget, the section could be stale. The failure mode
  is benign: `disenroll` tolerates an already-gone slot and converges.
- **Un-enroll is a de-escalation.** It only ever *removes* an unlock path, so the
  gate on it is for consistency and accident-prevention, not confidentiality. It is
  still gated because an accidental revocation costs the user real convenience and
  is only recoverable with the master password.
